# v1.7 MCP Server — 设计文档

## 目标

暴露 MCP 接口，让 AI 助手（Claude Code、Cursor、Windsurf 等）能访问和操作资料库，作为知识管理助手：读取资料、搜索内容、整理标签、撰写笔记。

## 决策汇总

| 决策项 | 选择 | 理由 |
|--------|------|------|
| 使用场景 | 知识管理（读写并重） | AI 不仅能查阅资料，还能主动整理 |
| 传输协议 | stdio | MCP 生态标准，所有客户端原生支持 |
| 数据访问 | 通过 HTTP 代理 | 统一数据访问入口，复用事件系统，避免 SQLite 并发问题 |
| 实现方案 | Node.js + @modelcontextprotocol/sdk | 官方 TS SDK 最成熟，项目已有 Node.js 环境 |
| 工具粒度 | 精简聚合（9 个工具） | 减少 AI 上下文占用，提高选择准确性 |
| 快照内容 | 纯文本提取，保存时写入 DB，分页读取 | 面向复用（MCP / v2.0 全文搜索 / AI 摘要） |

## 架构

```
Claude Code / Cursor / Windsurf
        │ stdin/stdout (MCP JSON-RPC)
        ▼
   shibei-mcp          ← Node.js 进程（@modelcontextprotocol/sdk）
        │ HTTP (Bearer token, 127.0.0.1:21519)
        ▼
   Tauri App (axum)    ← 主应用必须运行中
        │
        ▼
   SQLite + Storage
```

- MCP Server 是独立 Node.js 进程，由 AI 客户端管理生命周期（启动/停止）
- 通过 HTTP 调用主应用的 axum server，复用现有鉴权和事件系统
- 主应用未运行时，MCP 工具返回明确错误信息

## MCP 工具定义

### 读取类（6 个）

#### 1. `search_resources` — 搜索和浏览资料库

```
参数：
  query?: string          # 搜索关键词（>= 2 字符）
  folder_id?: string      # 限定文件夹
  tag_ids?: string[]      # 标签过滤（OR 逻辑）
  sort_by?: "created_at" | "annotated_at"
  sort_order?: "asc" | "desc"
返回：资料列表（id, title, url, description, folder_id, created_at）
```

所有参数可选。不传参即浏览全部资料。列表不含标签（避免 N+1 查询），需要标签时用 `get_resource` 查单条。

#### 2. `get_resource` — 获取单个资料详情

```
参数：
  resource_id: string
返回：元数据 + 标签列表
```

#### 3. `get_annotations` — 获取资料的高亮和评论

```
参数：
  resource_id: string
返回：
  highlights: [{id, text_content, color, created_at}]
  comments: [{id, highlight_id, content, created_at, updated_at}]
```

独立于 `get_resource`，避免评论内容过多时撑大响应。

#### 4. `get_resource_content` — 读取资料纯文本内容

```
参数：
  resource_id: string
  offset?: number         # 字符偏移量，默认 0
  max_length?: number     # 最大返回字符数，默认 50000
返回：
  content: string         # 纯文本片段
  total_length: number    # 全文总字符数
  has_more: boolean       # 是否有更多内容
```

从 DB `plain_text` 字段读取，支持分页。若 `plain_text` 为 NULL 且快照文件存在，则即时提取并回填 DB（懒填充）；若快照也不存在（未同步下载），返回提示 "Content not available. Please open this resource in Shibei app to download the snapshot first."。

#### 5. `list_folders` — 获取文件夹树

```
参数：无
返回：文件夹树结构（嵌套，含各文件夹资料计数）
```

复用现有 `GET /api/folders` + `GET /api/folder-counts`。

#### 6. `list_tags` — 获取所有标签

```
参数：无
返回：标签列表（id, name, color）
```

复用现有 `GET /api/tags`。AI 需要先调此工具获取 tag_id，才能在 `search_resources` 中按标签筛选或在 `manage_tags` 中添加/移除标签。

### 写入类（3 个）

#### 7. `update_resource` — 编辑资料元数据 / 移动文件夹

```
参数：
  resource_id: string
  title?: string          # 新标题
  description?: string    # 新描述
  folder_id?: string      # 目标文件夹 ID（移动）
```

传哪个改哪个，支持同时修改多个字段。

#### 8. `manage_tags` — 标签管理

```
参数：
  action: "create" | "add" | "remove"
  # action = "create" 时：
  name: string
  color: string           # 预设颜色：red, orange, yellow, green, cyan, blue, purple, pink
  # action = "add" | "remove" 时：
  resource_id: string
  tag_id: string
返回：
  # action = "create" 时返回 tag_id
```

工具描述中列出 8 种预设颜色值，帮助 AI 正确传参。

#### 9. `manage_notes` — 添加或编辑资料级笔记

```
参数：
  action: "create" | "update"
  # action = "create" 时：
  resource_id: string
  content: string         # Markdown 格式
  # action = "update" 时：
  comment_id: string
  content: string         # Markdown 格式
返回：comment_id
```

`create` 创建资料级笔记（`highlight_id` = null）。`update` 可编辑任意评论（包括高亮评论），AI 通过 `get_annotations` 获取 `comment_id`。

### 不暴露的操作

| 操作 | 原因 |
|------|------|
| 删除资料 / 文件夹 / 标签 | 破坏性操作，留给用户手动 |
| 创建高亮 | 需要 HTML anchor 定位，AI 无法精确计算 |
| 创建 / 删除文件夹 | 结构变更留给用户 |
| 回收站操作 | purge / restore 是管理操作 |
| 同步操作 | 内部机制，不对外暴露 |

## HTTP API 扩展

现有 API 只覆盖插件需求，MCP 需要新增以下路由。所有路由复用现有 Bearer token 中间件。

### 现有路由（无需改动）

| 路由 | 说明 | MCP 用途 |
|------|------|---------|
| `GET /api/folders` | 文件夹树 | `list_folders` |
| `GET /api/tags` | 标签列表 | `list_tags` |
| `GET /api/folder-counts` | 文件夹资料计数 | `list_folders` 补充 |

### 新增读取路由

| 路由 | 说明 |
|------|------|
| `GET /api/resources?folder_id=&tag_ids=&sort_by=&sort_order=&query=` | 资料列表（合并搜索 + 筛选 + 排序） |
| `GET /api/resources/:id` | 资料元数据 + 标签 |
| `GET /api/resources/:id/annotations` | 高亮 + 评论 |
| `GET /api/resources/:id/content?offset=0&max_length=50000` | 纯文本分页（含懒填充逻辑） |

### 新增写入路由

| 路由 | 说明 |
|------|------|
| `PUT /api/resources/:id` | 编辑资料元数据（title, description, folder_id），统一一个路由 |
| `POST /api/resources/:id/tags/:tag_id` | 为资料添加标签 |
| `DELETE /api/resources/:id/tags/:tag_id` | 移除资料标签 |
| `POST /api/tags` | 创建标签 |
| `POST /api/resources/:id/comments` | 添加资料级笔记（highlight_id = null） |
| `PUT /api/comments/:id` | 编辑评论内容 |

### 设计原则

- RESTful 风格，与现有路由一致
- 写操作完成后 emit 领域事件（`data:resource-changed` 等），前端自动刷新
- `PUT /api/resources/:id` 统一处理元数据编辑和文件夹移动，传哪个字段改哪个
- 聚合返回减少调用轮次（`GET /api/resources/:id` 含标签）

## 纯文本提取与存储

### 存储

`resources` 表新增 `plain_text` 字段（TEXT, nullable）。

选择 DB 字段而非独立文件的理由：
- 纯文本比源 HTML 小得多（去掉 base64 图片、CSS 等），几十 KB 级别
- 查询方便，未来可直接用于全文搜索
- 同步时跟随资料记录一起走

### 提取时机

1. **保存资料时**（HTTP `/api/save`）— 写入快照文件后立即提取纯文本写入 `plain_text`
2. **同步下载快照后** — 远端快照下载完成时补充提取
3. **MCP 读取时懒填充** — `GET /api/resources/:id/content` 若 `plain_text` 为 NULL 但快照文件存在，即时提取并回填 DB，后续请求直接读 DB

### 提取实现

Rust 侧新增 `extract_plain_text(html: &str) -> String`：
- 使用 `scraper` crate（基于 html5ever）解析 HTML
- 提取可见文本节点，过滤 `<script>`、`<style>`、`<noscript>` 等
- 保留段落结构（换行分隔）

### 复用场景

| 场景 | 用法 |
|------|------|
| v1.7 MCP | `/api/resources/:id/content` 从 `plain_text` 分页读取 |
| v2.0 全文搜索 | `search_index` 新增 content 列，数据源为 `plain_text` |
| v2.0 AI 摘要 | 直接读 `plain_text` 作为 LLM 输入 |

### 新增依赖

- Rust: `scraper`（HTML 解析，基于 html5ever + selectors）
- Node.js: `@modelcontextprotocol/sdk`（MCP 协议实现）

## Token 传递

主应用启动时将 Bearer token 写入 `{data_dir}/mcp-token`（macOS: `~/Library/Application Support/shibei/mcp-token`），纯文本，文件权限 600。主应用退出时删除该文件。路径通过 `dirs::data_dir()` 派生，跨平台兼容。

MCP 进程启动时读取此文件。文件不存在即判定主应用未运行。

MCP 进程获取 token 文件路径的方式：按平台约定硬编码（macOS: `~/Library/Application Support/shibei/mcp-token`），或通过环境变量 `SHIBEI_DATA_DIR` 覆盖（用于开发/测试）。

不使用环境变量或命令行参数传递 token 本身的原因：token 每次启动不同，不应硬编码在 MCP 客户端配置中。

## 错误处理

所有错误通过 MCP 协议正常返回（不 crash），让 AI 能理解并告知用户：

| 场景 | 错误信息 |
|------|---------|
| 主应用未运行（token 文件不存在 / HTTP 连接失败） | "Shibei app is not running. Please start the app first." |
| 资料不存在 | "Resource not found: {id}" |
| 快照未下载 | "Content not available. Please open this resource in Shibei app to download the snapshot first." |
| 参数无效 | "Invalid parameter: {detail}" |
| Token 失效（HTTP 401） | "Authentication failed. Token may be stale, please restart Shibei app." |

## 目录结构

```
mcp/
  package.json          # 独立包，不与前端 src/ 混合
  tsconfig.json
  src/
    index.ts            # 入口：创建 MCP Server，注册工具，启动 stdio transport
    client.ts           # HTTP 客户端：token 读取、请求封装、错误处理
    tools/
      search.ts         # search_resources
      resource.ts       # get_resource, get_resource_content, update_resource
      annotations.ts    # get_annotations, manage_notes
      folders.ts        # list_folders
      tags.ts           # list_tags, manage_tags
    types.ts            # 共享类型定义
```

### 依赖

- `@modelcontextprotocol/sdk` — MCP 协议实现
- `typescript` — 编译
- 无其他运行时依赖（HTTP 请求用 Node.js 原生 `fetch`）

### 构建

`tsc` 编译到 `mcp/dist/`，不需要 bundler。

## 数据库变更

### Migration 006

```sql
ALTER TABLE resources ADD COLUMN plain_text TEXT;
```

### 同步兼容

`plain_text` 是本地派生字段（从快照提取），不写入 `sync_log`，不参与同步传输。各设备独立提取：保存时提取、同步下载快照后提取、MCP 读取时懒填充。
