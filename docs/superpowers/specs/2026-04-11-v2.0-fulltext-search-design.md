# v2.0 — 快照全文搜索设计

## 目标

将快照 HTML 的正文纯文本纳入搜索索引，让用户可以搜索到网页正文内容，而不仅仅是标题、URL、标注等元数据。

## 背景

v1.4 第一期已建成完整的搜索基础设施：

- FTS5 虚拟表 `search_index`（trigram tokenizer），索引 title/url/description/highlights_text/comments_text
- `rebuild_search_index` / `delete_search_index` 维护管道，嵌入所有 CRUD 操作
- 前端搜索框（debounce 300ms，≥ 2 字符触发）
- `plain_text` 字段已存在（migration 006），保存资料时由 `extract_plain_text`（scraper crate）提取并写入 `resources.plain_text`

v1.4 设计文档已明确第二期方向："快照全文搜索：保存快照时提取 HTML 纯文本，写入同一套 FTS5 索引。复用第一期的全部基础设施。"

## 设计决策

| 决策 | 选择 | 理由 |
|------|------|------|
| 索引粒度 | 全文索引，不截断 | 个人资料库体量有限，全量覆盖更有价值 |
| 结果展示 | "正文匹配"标签，不显示 snippet | 改动小，满足"知道为什么搜到"的需求 |
| 老数据填充 | 启动时批量填充 | 确保所有资料可搜，不留盲区 |

## 后端设计

### 1. FTS5 Schema 迁移

FTS5 虚拟表不支持 `ALTER TABLE ADD COLUMN`。新建 migration `007_search_index_body.sql`：

```sql
DROP TABLE IF EXISTS search_index;

CREATE VIRTUAL TABLE search_index USING fts5(
    resource_id UNINDEXED,
    title,
    url,
    description,
    highlights_text,
    comments_text,
    body_text,
    tokenize='trigram'
);

-- 重置 FTS 初始化标志，启动时触发全量重建
DELETE FROM sync_state WHERE key = 'config:fts_initialized';
```

新增 `body_text` 列，内容来自 `resources.plain_text`。

### 2. `rebuild_search_index` 变更

在现有函数中增加读取 `plain_text` 的逻辑：

```rust
// 现有：读 title, url, description
// 新增：读 plain_text
let plain_text: Option<String> = conn.query_row(
    "SELECT plain_text FROM resources WHERE id = ?1 AND deleted_at IS NULL",
    params![resource_id],
    |row| row.get(0),
)?;

// INSERT 时增加 body_text 列
conn.execute(
    "INSERT INTO search_index (resource_id, title, url, description, highlights_text, comments_text, body_text)
     VALUES (?1, ?2, ?3, ?4, ?5, ?6, ?7)",
    params![..., plain_text.unwrap_or_default()],
)?;
```

合并到现有 SELECT 查询中，一次读取 title/url/description/plain_text，避免额外查询。

### 3. 启动时批量填充 `plain_text`

在 FTS 全量重建之前，先回填缺失的 `plain_text`。修改 `lib.rs` 中的启动逻辑：

```
启动线程:
  1. 检查 is_fts_initialized
  2. 如果未初始化:
     a. 扫描 plain_text IS NULL 且 deleted_at IS NULL 的资料
     b. 逐条：读 snapshot.html → extract_plain_text → set_plain_text
        （best-effort：读文件失败则跳过，不阻塞其他资料）
     c. rebuild_all_search_index
     d. mark_fts_initialized
```

回填逻辑需要访问 storage 路径（`{data_dir}/storage/{resource_id}/snapshot.html`）。当前 FTS 初始化线程只捕获了 `fts_pool`（DB 连接池），需要额外 clone `data_dir: PathBuf` 传入线程闭包。

### 4. `search_resources` 查询变更

#### FTS MATCH 路径（≥ 3 字符）

无需改动。FTS5 MATCH 自动搜索所有已索引列（包括新增的 `body_text`）。

#### LIKE fallback 路径（< 3 字符）

包含 `body_text`。虽然对全文做 `LIKE %xx%` 是全表扫描，但个人资料库体量有限（数百条级别），实测可接受。不包含的话 2 字中文词（如"胜利"、"伊朗"）将无法搜索正文：

```sql
AND (si.title LIKE ?1 OR si.url LIKE ?1 OR si.description LIKE ?1
     OR si.highlights_text LIKE ?1 OR si.comments_text LIKE ?1
     OR si.body_text LIKE ?1)
```

### 5. 匹配来源标记

搜索命令返回值中需标记"这条结果是否因正文匹配而命中"。方案：

**返回值增加 `matched_body: bool` 字段**：

在 `search_resources` 函数中，对每条结果做一次轻量检查：资料的 title/url/description/highlights/comments 是否包含查询词。如果都不包含，则说明命中来自 body_text。

具体实现：不额外查 FTS，而是在 Rust 层拿到 `Resource` 后对 title/url/description 做大小写不敏感的 `contains` 检查。highlights_text 和 comments_text 从 search_index 表 JOIN 读取后同样检查。如果所有元数据字段都不包含查询词，则 `matched_body = true`。

**返回类型变更**：

```rust
#[derive(Serialize)]
#[serde(rename_all = "camelCase")]
pub struct SearchResult {
    #[serde(flatten)]
    pub resource: Resource,
    pub matched_body: bool,
}
```

使用 `#[serde(flatten)]` 将 `Resource` 字段平铺到 `SearchResult` 中，前端接收到的 JSON 是 `Resource` 的所有字段加上 `matchedBody`（camelCase）。这样前端只需在现有 `Resource` 类型上扩展，不需要嵌套解构。

`cmd_search_resources` 返回 `Vec<SearchResult>` 而非 `Vec<Resource>`。

前端 TypeScript 类型同步更新。

### 6. 不改动的部分

- `extract_plain_text` 提取逻辑不变
- 保存资料时写 `plain_text` 的流程不变（server/mod.rs 中已有）
- 同步 apply 后提取 `plain_text` 的流程不变（sync/engine.rs 中已有）
- CRUD 操作中调用 `rebuild_search_index` 的触发时机不变
- MCP 搜索接口：HTTP search handler（`server/mod.rs`）需适配新的 `SearchResult` 返回类型。MCP server（`mcp/src/tools/search.ts`）调用 HTTP 端点，响应 JSON 结构变更后需同步更新 `mcp/src/types.ts` 解析逻辑。但 MCP 工具返回纯文本格式化结果，只需在格式化时忽略 `matched_body` 字段即可，改动量极小

## 前端设计

### 搜索结果"正文匹配"标签

当 `matched_body === true` 时，在资料卡片上显示标签：

```
┌──────────────────────────────────┐
│  深度学习入门指南        [正文匹配] │  ← 标签
│  example.com · 2026-04-10       │
└──────────────────────────────────┘
```

- 标签样式：小号文字 + 浅色背景（类似现有 tag 样式），不抢夺视觉焦点
- 仅在搜索模式下且 `matched_body` 为 true 时显示
- 标题/URL 命中时不显示此标签（因为标题高亮已经说明了匹配原因）
- i18n：`search:bodyMatch`，中文"正文匹配"，英文"Body match"

### 组件变更

| 组件 | 变更 |
|------|------|
| `ResourceList.tsx` | `DraggableResourceItem` 接收 `matchedBody` prop，条件渲染标签 |
| `ResourceList.module.css` | 新增 `.bodyMatchTag` 样式 |
| `useResources.ts` | 搜索模式下解析 `SearchResult` 类型，维护 `matchedBodyMap: Map<string, boolean>` |

### 类型变更

`src/types/` 或 `src/lib/commands.ts` 新增：

```typescript
interface SearchResult extends Resource {
  matchedBody: boolean;
}
```

## 数据流

```
用户输入搜索词
    ↓
ResourceList（debounce 300ms）
    ↓
useResources → cmd_search_resources(query, folderId, tagIds, sortBy, sortOrder)
    ↓
Rust: FTS5 MATCH (title + url + description + highlights + comments + body_text)
    → JOIN resources → 过滤 folder/tags → 排序
    → 对每条结果判断 matched_body
    ↓
返回 Vec<SearchResult>
    ↓
ResourceList 渲染：
  - 标题高亮（现有逻辑不变）
  - matched_body=true 时显示"正文匹配"标签
```

## 启动时回填流程

```
应用启动
    ↓
后台线程检查 config:fts_initialized
    ↓ (未初始化)
查询 SELECT id FROM resources
  WHERE plain_text IS NULL AND deleted_at IS NULL
    ↓
逐条处理:
  读取 {data_dir}/storage/{id}/snapshot.html
  → extract_plain_text(html)
  → set_plain_text(conn, id, text)
  （失败跳过，打印 warn 日志）
    ↓
rebuild_all_search_index (含 body_text)
    ↓
mark_fts_initialized
```

对于新安装用户（无历史数据），此流程在数毫秒内完成。
对于有大量历史资料的用户，回填涉及磁盘 I/O（读 HTML 文件），但只执行一次。
