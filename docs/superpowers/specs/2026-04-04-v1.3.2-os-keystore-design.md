# v1.3.2 — OS 安全存储记住加密密钥 设计文档

## 目标

E2EE 主密钥（MK）每次应用启动后丢失，用户需重新输入密码解锁。本版本将 MK 存入操作系统安全存储（macOS Keychain / Windows Credential Manager / Linux Secret Service），实现启动时自动解锁。

## 设计决策

### 存 MK 而非密码

将解密后的 **MK（32 字节）** 存入 OS keychain，而非用户密码。原因：

1. MK 是实际需要的值，避免每次启动重复执行 Argon2id 派生（约 200ms+）
2. 不暴露用户可能复用的密码——即使 keychain 被攻破，攻击者只拿到 MK
3. 修改密码后 MK 不变，keychain 无需更新

### Opt-in 设计

默认不记住。用户在加密已解锁状态下主动开启 toggle 才存入 keychain。

### 依赖选型

新增 `keyring` crate（[crates.io/crates/keyring](https://crates.io/crates/keyring)）：
- macOS → Keychain Services
- Windows → Credential Manager
- Linux → Secret Service (D-Bus)
- 轻量（仅系统 API 绑定），维护活跃

**Feature flags**：使用 `keyring = { version = "3", features = ["apple-native"] }` 仅启用 macOS 后端，减小依赖树。未来如需跨平台，再切换回默认 features。

## 新增模块

### `src-tauri/src/sync/os_keystore.rs`

三个函数，封装 `keyring` crate 的调用：

```rust
use zeroize::Zeroizing;

const SERVICE: &str = "shibei";
const ACCOUNT: &str = "encryption-master-key";

pub enum KeystoreError {
    NotFound,
    PlatformError(String),
    InvalidFormat(String),
}

/// 将 MK 存入 OS keychain（原始 32 字节，通过 set_secret）。
pub fn save_master_key(mk: &[u8; 32]) -> Result<(), KeystoreError>

/// 从 OS keychain 读取 MK。不存在返回 Ok(None)。
/// 使用 get_secret 直接获取原始字节，无需 base64 中间值。
pub fn load_master_key() -> Result<Option<Zeroizing<[u8; 32]>>, KeystoreError>

/// 删除 OS keychain 中的 MK 条目。不存在时静默成功。
pub fn delete_master_key() -> Result<(), KeystoreError>
```

**内存安全**：使用 `keyring` crate 的 `set_secret(&[u8])`/`get_secret() → Vec<u8>` API 直接存取原始字节，跳过 base64 编解码，消除中间值泄露风险。`get_secret` 返回的 `Vec<u8>` 在复制到 `Zeroizing<[u8; 32]>` 后立即 `zeroize()`。

## 新增 Tauri Commands

### `cmd_set_remember_key(remember: bool)`

**前置条件**：`remember=true` 时要求 `EncryptionState` 已解锁，否则返回错误。

```
if remember:
    mk = encryption_state.get_key()  // 必须 Some
    os_keystore::save_master_key(&mk)
    sync_state::set("config:remember_encryption_key", "true")
else:
    os_keystore::delete_master_key()  // 忽略 NotFound
    sync_state::delete("config:remember_encryption_key")
emit(DATA_CONFIG_CHANGED, { scope: "encryption" })
```

### `cmd_get_remember_key() → bool`

读 DB flag `config:remember_encryption_key`，返回 `true` 或 `false`。

### `cmd_auto_unlock() → AutoUnlockResult`

启动时由前端调用，尝试从 keychain 恢复 MK：

```rust
#[derive(Serialize)]
#[serde(rename_all = "snake_case")]
pub enum AutoUnlockResult {
    Unlocked,           // 成功，MK 已验证并写入 EncryptionState
    UnlockedUnverified, // MK 已写入 EncryptionState，但 S3 不可达未完成验证（乐观信任）
    NoStoredKey,        // keychain 中无 MK（首次或已关闭记住）
    KeychainError,      // 系统钥匙串不可用
    KeyMismatch,        // MK 已失效（被其他设备重置加密）
}
```

**流程**：

```
1. os_keystore::load_master_key()
   → Err(PlatformError) → return KeychainError
   → Ok(None) → return NoStoredKey
   → Ok(Some(mk)) → 继续

2. 验证 MK 有效性（防止其他设备重置加密后 MK 过期）：
   build_raw_s3_backend()
   → 成功：
       download("meta/keyring.json")
       → 成功：
           解析 keyring，compute_verification_hash(mk) vs keyring.verification_hash
           → 匹配：encryption_state.set_key(mk), return Unlocked
           → 不匹配：os_keystore::delete_master_key(),
                     sync_state::delete("config:remember_encryption_key"),
                     return KeyMismatch
       → 下载失败：encryption_state.set_key(mk), return UnlockedUnverified
   → 失败（无 S3 配置等）：
       encryption_state.set_key(mk), return UnlockedUnverified
```

**离线策略**：S3 不可达时乐观信任 keychain 中的 MK，返回 `UnlockedUnverified` 让前端知道已解锁但未完成验证。MK 被替换是极端场景（同一用户在另一台设备重置加密），而离线使用是常见场景。sync 时如果 MK 确实失效，AEAD 校验会报错，用户可手动重新输入密码。

## 修改现有 Commands

### `cmd_unlock_encryption` — 修复同步重置副作用

**问题**：当前每次调用都重置同步进度（清除 `sync_log.uploaded`、删除 `remote:*`、清除 `last_sync_at`）。这是为"新设备首次解锁"设计的，但在已有设备重启后手动解锁时会导致不必要的全量重同步。v1.3.2 引入 auto_unlock 后，偶尔的手动解锁触发重置会让用户困惑。

**修改**：只在首次解锁（本地无 `config:encryption_enabled`）时重置同步进度。

```rust
// 修改后的 cmd_unlock_encryption 逻辑：
let is_first_unlock = sync_state::get(&conn, "config:encryption_enabled")?
    .map(|v| v == "true")
    .unwrap_or(false) == false;

encryption_state.set_key(mk);
sync_state::set(&conn, "config:encryption_enabled", "true")?;

if is_first_unlock {
    // 新设备首次解锁：重置同步进度，触发 Phase 1.5
    conn.execute("UPDATE sync_log SET uploaded = 0", [])?;
    let remote_keys = sync_state::list_by_prefix(&conn, "remote:")?;
    for (key, _) in &remote_keys {
        sync_state::delete(&conn, key)?;
    }
    sync_state::delete(&conn, "last_sync_at")?;
}

// 如果用户开启了记住密钥，自动存入 keychain
if sync_state::get(&conn, "config:remember_encryption_key")?
    .map(|v| v == "true")
    .unwrap_or(false)
{
    let _ = os_keystore::save_master_key(&mk); // 失败不阻断
}

emit(DATA_CONFIG_CHANGED, { scope: "encryption" });
```

### `cmd_setup_encryption` — 清理旧 keychain 条目

在生成新 keyring 之前，先清除 keychain 中可能存在的旧 MK：

```rust
// 在 cmd_setup_encryption 开头新增：
let _ = os_keystore::delete_master_key(); // 忽略 NotFound
```

这处理了两个场景：
1. 用户在同一设备重新设置加密 → 旧 MK 失效
2. 用户禁用后重新启用加密 → 旧 MK 失效

### `useSync` hook — 自动解锁

新增 `autoUnlockPending` 状态，防止 auto_unlock 期间 UI 闪烁（锁图标闪一下再消失）：

```typescript
const [autoUnlockPending, setAutoUnlockPending] = useState(false);

useEffect(() => {
  cmd.getEncryptionStatus().then(async (es) => {
    setEncryptionEnabled(es.enabled);

    // 如果加密已启用但未解锁，尝试自动解锁
    if (es.enabled && !es.unlocked) {
      setAutoUnlockPending(true);  // 进入中间态，SyncStatus 显示"正在检查..."
      try {
        const result = await cmd.autoUnlockEncryption();
        switch (result) {
          case "unlocked":
          case "unlocked_unverified":
            setEncryptionUnlocked(true);
            break;
          case "keychain_error":
            toast("系统钥匙串不可用，请手动输入密码", { icon: "ℹ️" });
            break;
          case "key_mismatch":
            toast("加密密钥已变更，请重新输入密码", { icon: "⚠️" });
            break;
          case "no_stored_key":
            break; // 正常情况，不提示
        }
      } finally {
        setAutoUnlockPending(false);
      }
    } else {
      setEncryptionUnlocked(es.unlocked);
    }
  }).catch(() => {});
}, []);

// 返回值新增 autoUnlockPending
return { ..., autoUnlockPending };
```

**SyncStatus 使用 `autoUnlockPending`**：当 `autoUnlockPending=true` 时，不显示锁图标，而是显示"正在检查..."或 spinner，避免"需解锁"闪烁。

const [rememberKey, setRememberKey] = useState(false);

    cmd.getRememberKey().then(setRememberKey).catch(() => {});

## Keychain 清理矩阵

确保所有会使 MK 失效的操作都清理 keychain：

| 操作 | Keychain 处理 | DB flag 处理 |
|------|--------------|-------------|
| `cmd_setup_encryption` | `delete_master_key()`（开头执行） | 不变 |
| `cmd_set_remember_key(false)` | `delete_master_key()` | 删除 flag |
| `cmd_auto_unlock` → KeyMismatch | `delete_master_key()` | 删除 flag |
| `cmd_change_encryption_password` | 不处理（MK 不变） | 不变 |
| `cmd_unlock_encryption` + remember=true | `save_master_key()` | 不变 |

## 安全考量

1. **存 MK 不存密码** — keychain 泄露不暴露用户可能复用的密码
2. **OS 保护** — macOS Keychain 受用户登录密码/TouchID 保护；Windows Credential Manager 受用户账户保护
3. **显式 opt-in** — 默认不记住，用户主动开启
4. **Zeroizing** — 从 keychain 读取的 base64 String 在 decode 后立即 zeroize；MK 始终用 `Zeroizing<[u8; 32]>` 包装
5. **MK 验证** — auto_unlock 后从 S3 下载 keyring.json 验证 hash，防止过期 MK 导致同步失败
6. **降级提示** — keychain 不可用或 MK 失效时 toast 告知用户原因，而非静默回退
7. **macOS 授权** — UI 说明文字提及"可能需要系统密码或 TouchID 验证"

---

## 不在范围

- 生物识别解锁（TouchID/FaceID 直接解锁，无需密码）
- 自动锁定（idle timeout 后清除内存中的 MK）
- 本地数据加密
- "禁用加密"独立命令（当前不存在此功能）
