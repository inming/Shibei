# v1.3 — S3 云同步设计文档

## 目标

实现多设备双向数据同步，让资料库不再局限于单机。基于 S3 兼容存储，支持 AWS S3、MinIO、Cloudflare R2、阿里云 OSS 等。

## 核心设计决策

| 维度 | 决策 | 理由 |
|------|------|------|
| 同步模式 | 多设备双向同步 | 日常在多台设备使用 |
| 冲突策略 | Last-Write-Wins (LWW) | 个人资料库并发编辑概率极低，LWW 实现简单可靠 |
| 排序时钟 | Hybrid Logical Clock (HLC) | 容忍多设备时钟偏差，保证全序 |
| 触发方式 | 手动触发 + 启动时自动同步一次 | 覆盖最常见场景，不引入定时器复杂度 |
| 数据范围 | 元数据全量同步 + 快照按需下载 | snapshot 文件可能很大，按需拉取兼顾效率和完整性 |
| 加密 | v1.3 不做 E2EE，明文存储 | 先跑通同步协议，v1.3.1 叠加加密层 |
| S3 兼容性 | 支持自定义 endpoint | 几乎零成本，灵活度大 |
| 凭据存储 | Stronghold / keyring 加密存储 | 不明文存储 access key/secret key |

## 一、数据模型变更

### 1.1 现有表添加字段

6 张业务表（folders、resources、tags、highlights、comments、resource_tags）均添加：

```sql
ALTER TABLE {table} ADD COLUMN hlc TEXT;
ALTER TABLE {table} ADD COLUMN deleted_at TEXT;
```

- `hlc`：Hybrid Logical Clock 时间戳，同步专用，格式 `"{wall_ms}-{counter:04}-{device_id}"`，字典序可比较
- `deleted_at`：软删除标记，ISO8601 时间戳，NULL 表示未删除
- `created_at` / `updated_at` 保持 ISO8601 不变，业务逻辑（排序、展示）继续使用

**resource_tags 特殊处理**：resource_tags 作为 resource 的附属数据同步——打/摘标签时更新 resource 的 hlc，sync_log payload 中包含完整 tag_ids 列表。resource_tags 表的 hlc/deleted_at 字段仅用于本地软删除标记，不独立产生 sync_log 记录。

**现有数据迁移**：hlc 初始值从 `updated_at` 转换生成（wall_time_ms + counter=0 + device_id）。

### 1.2 新增 sync_log 表

```sql
CREATE TABLE sync_log (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  entity_type TEXT NOT NULL,  -- folder / resource / tag / highlight / comment
  entity_id TEXT NOT NULL,
  operation TEXT NOT NULL,    -- INSERT / UPDATE / DELETE
  payload TEXT NOT NULL,      -- 变更后完整实体 JSON（DELETE 时存删除前数据 + deleted_at）
  hlc TEXT NOT NULL,          -- 本条变更的 HLC 时间戳
  device_id TEXT NOT NULL,
  uploaded INTEGER NOT NULL DEFAULT 0  -- 0=未上传, 1=已上传
);
CREATE INDEX idx_sync_log_uploaded ON sync_log(uploaded);
```

**触发方式**：在每个 CRUD 函数中，操作成功后追加一条 sync_log。不用 SQLite trigger。

**payload 内容**：

- INSERT/UPDATE：变更后的完整实体 JSON。resource 类型额外包含 `tag_ids` 数组。
- DELETE：删除前的实体数据 + `deleted_at` 时间戳。

### 1.3 新增 sync_state 表

```sql
CREATE TABLE sync_state (
  key TEXT PRIMARY KEY,
  value TEXT NOT NULL
);
```

存储内容：

| key | value | 说明 |
|-----|-------|------|
| `last_uploaded_log_id` | `"142"` | 本地已上传到 S3 的最新 sync_log id |
| `remote:<device_id>:last_seq` | `"2026-04-02T..."` | 该远端设备已处理到的最新 seq |
| `snapshot:<resource_id>` | `"pending"` / `"synced"` | 快照文件下载状态 |
| `last_sync_at` | ISO8601 | 最后一次同步完成时间 |
| `config:s3_endpoint` | URL | S3 endpoint（可选，空字符串表示默认 AWS） |
| `config:s3_region` | string | S3 region |
| `config:s3_bucket` | string | S3 bucket name |

非敏感 S3 配置统一存储在 sync_state 表中（`config:` 前缀），不再使用单独的 sync_config.json 文件。敏感凭据（access_key、secret_key）存储在 Stronghold/keyring 中。

### 1.4 软删除改造

- 删除操作改为 `UPDATE SET deleted_at = now(), hlc = new_hlc WHERE id = ?`
- **所有查询**加 `WHERE deleted_at IS NULL` 条件
- 物理清理：`deleted_at` 超过 90 天的记录直接物理删除，在 compaction 时顺便执行
- 90 天未同步的设备视为需要全量重建

## 二、Hybrid Logical Clock (HLC)

### 2.1 数据结构

```rust
pub struct HLC {
    pub wall_time_ms: i64,  // 物理时钟，毫秒
    pub counter: u32,       // 逻辑计数器
    pub device_id: String,  // 设备标识
}
```

序列化为字符串：`"1711987200000-0003-a1b2c3d4-e5f6-7890-abcd-ef1234567890"`，字典序可比较。

### 2.2 推进规则

**本地事件**（写操作时）：

```
new.wall_time_ms = max(now_ms(), last_hlc.wall_time_ms)
if new.wall_time_ms == last_hlc.wall_time_ms:
    new.counter = last_hlc.counter + 1
else:
    new.counter = 0
new.device_id = local_device_id
```

**接收远端事件**（apply 时）：

```
new.wall_time_ms = max(now_ms(), local_hlc.wall_time_ms, remote_hlc.wall_time_ms)
if new.wall_time_ms == local_hlc.wall_time_ms == remote_hlc.wall_time_ms:
    new.counter = max(local_hlc.counter, remote_hlc.counter) + 1
elif new.wall_time_ms == local_hlc.wall_time_ms:
    new.counter = local_hlc.counter + 1
elif new.wall_time_ms == remote_hlc.wall_time_ms:
    new.counter = remote_hlc.counter + 1
else:
    new.counter = 0
new.device_id = local_device_id
```

### 2.3 比较规则

字典序比较 HLC 字符串：先比 wall_time_ms → 再比 counter → 最后比 device_id。保证全序，没有平局。

## 三、设备标识

- 首次启动同步功能时生成 `device_id`（完整 UUID v4，如 `"a1b2c3d4-e5f6-7890-abcd-ef1234567890"`）
- 持久化到 `<app_data_dir>/device_id` 文件
- sync_log 每条记录、HLC 中均包含 device_id
- 同步时跳过自身 device_id 的远端变更

## 四、S3 存储层

### 4.1 存储后端抽象

```rust
#[async_trait]
pub trait SyncBackend: Send + Sync {
    async fn upload(&self, key: &str, data: &[u8]) -> Result<String>;  // 返回 ETag
    async fn download(&self, key: &str) -> Result<Vec<u8>>;
    async fn list(&self, prefix: &str) -> Result<Vec<ObjectInfo>>;
    async fn delete(&self, key: &str) -> Result<()>;
    async fn head(&self, key: &str) -> Result<Option<ObjectMeta>>;
}

pub struct ObjectInfo {
    pub key: String,
    pub size: u64,
    pub last_modified: String,
}

pub struct ObjectMeta {
    pub etag: String,
    pub size: u64,
}
```

### 4.2 S3 实现

使用 `rust-s3` crate（轻量，支持自定义 endpoint）。

配置项：
- `endpoint`（可选，默认 AWS）
- `region`
- `bucket`
- `access_key`（Stronghold/keyring 存储）
- `secret_key`（Stronghold/keyring 存储）

### 4.3 S3 数据布局

```
<bucket>/
  devices/
    <device_id>.json            -- 设备注册信息 { device_id, name, first_seen }
  sync/
    <device_id>/
      <seq_id>.jsonl            -- 增量变更日志批次
  snapshots/
    <resource_id>/
      snapshot.html             -- 快照文件
  state/
    snapshot-<timestamp>.json   -- 全量状态快照（所有实体当前状态）
    compaction-pending.json     -- 待清理文件列表（两阶段清理用）
```

### 4.4 凭据存储

**首选**：`tauri-plugin-stronghold`（Tauri 官方，基于 IOTA Stronghold，跨平台加密存储）。

**备选**：`keyring` crate（macOS → Keychain，Linux → Secret Service，Windows → Credential Manager）。实现时先验证 Stronghold 对 Tauri 2.x 兼容性，不兼容则切 keyring。

非敏感配置（endpoint、region、bucket）存储在 sync_state 表中（见 1.3 节），不使用单独的配置文件。

## 五、同步协议

### 5.1 同步流程

每次 sync 执行（手动触发或启动时自动）：

```
Phase 1: 上传本地变更
  1. 读取 sync_log WHERE uploaded = 0，按 id 排序
  2. 打包为 JSONL 文件（每行一条 sync_log 记录的 JSON）
  3. 上传到 sync/<device_id>/<seq_id>.jsonl
     seq_id = 当前时间戳，保证唯一且有序
  4. 上传成功后标记 uploaded = 1，更新 last_uploaded_log_id
  5. 上传新增资料的 snapshot.html 到 snapshots/<resource_id>/

Phase 2: 拉取远端变更
  1. 列出 sync/ 下所有 device 目录（排除自身 device_id）
  2. 对每个远端设备，列出其 JSONL 文件
  3. 根据 sync_state 中 remote:<device_id>:last_seq 跳过已处理文件
  4. 下载未处理的 JSONL 文件，按文件名（时间戳）排序

Phase 3: Apply 远端变更
  1. 解析所有变更记录
  2. 按 (operation, entity_type) 拓扑排序：
     - INSERT/UPDATE：folder → tag → resource（含 resource_tags）→ highlight → comment
     - DELETE：comment → highlight → resource（含 resource_tags）→ tag → folder
     注意：resource 和 tag 都必须先于 resource_tags 存在，
     因此 tag INSERT 排在 resource 前面。全量快照导入同样遵循此顺序。
  3. 逐条 apply（见 5.2 Apply 逻辑）
  4. 更新 sync_state 中各设备的 last_seq

Phase 4: 通知前端
  1. 发送 Tauri Event: sync-completed { changed_entity_types }
  2. 前端根据 changed_entity_types 刷新对应视图
```

### 5.2 Apply 逻辑（LWW + 幂等）

```
对于每条远端变更记录 (entity_type, entity_id, operation, payload, hlc):

  跳过 device_id == local_device_id 的记录

  CASE INSERT:
    本地查找 entity_id（含软删除）：
    - 不存在 → 插入（标记 snapshot 为 pending 如果是 resource）
    - 存在且未删除 → 比较 hlc，远端更新则覆盖
    - 存在且已删除 → 比较 hlc，远端更新则恢复（清除 deleted_at）

  CASE UPDATE:
    本地查找 entity_id：
    - 不存在 → 视为 INSERT 处理
    - 存在 → 比较 hlc，远端更新则覆盖
    对 resource 类型：额外 diff tag_ids 列表，增删 resource_tags

  CASE DELETE:
    本地查找 entity_id：
    - 存在且未删除 → 比较 hlc，远端更新则软删除
    - 存在且已删除 → 忽略（已删）
    - 不存在 → 忽略
```

**幂等保证**：基于 entity_id 主键 + hlc 比较。同一条变更重复 apply，hlc 相同则跳过覆盖，不会产生副作用。

### 5.3 快照按需下载

- 远端同步来的 resource 在 sync_state 中标记 `snapshot:<resource_id> = pending`
- 打开阅读器时检查该标记，为 pending 则：
  1. 从 `snapshots/<resource_id>/snapshot.html` 下载
  2. 存入本地 `storage/<resource_id>/snapshot.html`
  3. 更新标记为 synced
- UI 中标记未下载的资料（云图标），阅读器打开时显示下载进度

### 5.4 Compaction

**触发条件**：每次同步时检查，当某个设备目录下 JSONL 文件数 > 100 或总大小 > 10MB。

**流程**：

```
1. 导出本地数据库全量状态为 JSON（所有实体当前状态，含 deleted_at 标记）
2. 上传到 state/snapshot-<timestamp>.json
3. 读取 compaction-pending.json（上次待清理列表）
4. 删除上次标记的 JSONL 文件（两阶段：上次标记的这次删）
5. 将本次要清理的 JSONL 文件列表写入 compaction-pending.json
6. 物理清理 deleted_at 超过 90 天的本地记录
```

**新设备首次同步**：

```
1. 下载最新 state/snapshot-*.json 全量导入
2. 回放该快照时间戳之后的增量 JSONL 文件
3. 所有远端 resource 的 snapshot 标记为 pending（按需下载）
```

## 六、同步锁

使用 `tokio::sync::Mutex` 的 `try_lock()` 实现 RAII 锁：

```rust
// sync_engine.rs
let _guard = match self.sync_lock.try_lock() {
    Ok(guard) => guard,
    Err(_) => {
        log::info!("Sync already in progress, skipping");
        return Ok(SyncResult::Skipped);
    }
};
// ... 执行同步 ...
// guard drop 时自动释放
```

- 手动触发和启动自动触发共用同一把锁，不会重叠
- 进程被 kill 后锁随进程消亡，无残留问题

## 九、错误处理

| 场景 | 处理方式 |
|------|---------|
| S3 连接失败 | 记录错误，前端显示"同步失败"，不阻塞应用使用 |
| 上传中断 | 下次同步重传（未标记 uploaded 的记录重新打包） |
| 下载中断 | 下次同步重新下载（基于 last_seq 断点续传） |
| Apply 失败（单条） | 记录错误日志，跳过该条，继续处理其余变更 |
| 快照下载失败 | 保持 pending 状态，用户可重试或下次同步时再试 |
| 凭据读取失败 | 提示用户重新配置 S3 凭据 |

## 十一、数据库 Migration

新增 migration 003：

```sql
-- 003_sync.sql

-- 业务表添加同步字段
ALTER TABLE folders ADD COLUMN hlc TEXT;
ALTER TABLE folders ADD COLUMN deleted_at TEXT;
ALTER TABLE resources ADD COLUMN hlc TEXT;
ALTER TABLE resources ADD COLUMN deleted_at TEXT;
ALTER TABLE tags ADD COLUMN hlc TEXT;
ALTER TABLE tags ADD COLUMN deleted_at TEXT;
ALTER TABLE highlights ADD COLUMN hlc TEXT;
ALTER TABLE highlights ADD COLUMN deleted_at TEXT;
ALTER TABLE comments ADD COLUMN hlc TEXT;
ALTER TABLE comments ADD COLUMN deleted_at TEXT;
ALTER TABLE resource_tags ADD COLUMN hlc TEXT;
ALTER TABLE resource_tags ADD COLUMN deleted_at TEXT;

-- 同步日志
CREATE TABLE sync_log (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  entity_type TEXT NOT NULL,
  entity_id TEXT NOT NULL,
  operation TEXT NOT NULL,
  payload TEXT NOT NULL,
  hlc TEXT NOT NULL,
  device_id TEXT NOT NULL,
  uploaded INTEGER NOT NULL DEFAULT 0
);
CREATE INDEX idx_sync_log_uploaded ON sync_log(uploaded);
CREATE INDEX idx_sync_log_entity ON sync_log(entity_type, entity_id);

-- 同步状态
CREATE TABLE sync_state (
  key TEXT PRIMARY KEY,
  value TEXT NOT NULL
);
```

**现有数据的 hlc 初始化**：在 migration SQL 中同步完成，不依赖应用层异步填充。migration 执行时 device_id 尚未生成（首次配置同步时才生成），因此使用固定占位符 `"pre-sync"` 作为 device_id：

```sql
UPDATE folders SET hlc = strftime('%s', updated_at) * 1000 || '-0000-pre-sync' WHERE hlc IS NULL;
UPDATE resources SET hlc = strftime('%s', updated_at) * 1000 || '-0000-pre-sync' WHERE hlc IS NULL;
UPDATE tags SET hlc = strftime('%s', updated_at) * 1000 || '-0000-pre-sync' WHERE hlc IS NULL;
UPDATE highlights SET hlc = strftime('%s', created_at) * 1000 || '-0000-pre-sync' WHERE hlc IS NULL;
UPDATE comments SET hlc = strftime('%s', updated_at) * 1000 || '-0000-pre-sync' WHERE hlc IS NULL;
```

这些 `pre-sync` HLC 值在首次同步上传时会被任何真实设备的 HLC 覆盖（LWW 比较时 `pre-sync` 字典序小于任何 UUID），行为正确。
