# Phase 4.1 — S3 凭据内存化 + 桌面 OS Keychain

**Status**: shipped (2026-04-23)

**Goal**: 把 S3 `access_key` / `secret_key` 从 SQLite 明文搬到 (a) 鸿蒙的内存 RwLock、(b) 桌面 OS 钥匙串，关闭 Phase 4 遗留的 "kill 后自动失效的 SQLite runtime cache" 这一中间形态。

**Outcome**: 抢到手机或桌面 DB 文件 (`shibei.db`) 的攻击者不再直接拿到 S3 凭据；想跑 sync 必须先过 HUKS（手机）或系统钥匙串（桌面）。两端统一了"凭据不落地 SQLite"的威胁模型。

---

## 1. 背景

Phase 4 (`2026-04-22-phase4-lockscreen-huks-design.md`) 落地时,出于"一次性改动边界可控"的考虑,**保留了** SQLite `sync_state:config:s3_access_key/secret_key` 作为 runtime cache——即便手机 `secure/s3_creds.blob` 已经是 HUKS wrap,冷启动时 `primeS3Creds` 解完 HUKS 仍然把明文写回 SQLite 给 `build_raw_backend` 读。Phase 4 的交付文档里明写了"下一期 Phase 4.1 砍掉"。

**触发收紧的**:

1. **威胁模型一致性**:攻击路径 `adb/hdc pull shibei.db` 仍然能直接拿到 AK/SK——哪怕 kill app 后 SQLite 里的行还在(靠 `reset_device` 或下次 `primeS3Creds` 覆写才清)。Phase 4 只关闭了冷态(`secure/*.blob`)的明文读,kill 态 SQLite 依然裸奔。
2. **桌面未对齐**:桌面端 `credentials::store_credentials` / `load_credentials` 一直走 SQLite;macOS 拖走 `~/Library/Application Support/shibei/shibei.db` 就能拿 AK/SK。移动端 Phase 4 落地后,桌面反而成了安全短板。

---

## 2. 威胁模型

### 2.1 本次收紧的威胁

| 威胁 | Phase 4 | Phase 4.1 |
|---|---|---|
| `hdc file recv shibei.db`(手机 kill 态) | SQLite 有明文 AK/SK | SQLite 不写 AK/SK,RwLock 随进程死 |
| 盗走 Mac `shibei.db` 文件 | 明文 AK/SK | 无 AK/SK;攻击者需同时拿到钥匙串解锁密码 |
| 进程内存 dump(手机 / 桌面) | N/A——凡是跑中 sync 都会有明文 | **不变**;这是运行时攻击,本期不防 |

### 2.2 本次不解决

- 进程内存被 root 权限 dump → AK/SK 明文出现在 `Arc<RwLock>` 里(手机)或 `Credentials` struct 栈上(桌面)——这是运行时攻击,超出应用层
- macOS 用户把钥匙串密码设成弱密码 / 关闭自动锁定 → 本来就是系统级假设
- 桌面 Linux 无 secret-service daemon 环境 → 降级到 SQLite fallback,安全模型回到 Phase 4 水平(见 §4.3)

## 3. 手机端:S3 creds → `Arc<RwLock<Option<(String, String)>>>`

### 3.1 数据流

```
冷启动:
  EntryAbility.onWindowStageCreate
    → state::init()
    → ShibeiService.primeS3Creds()
        ├── s3CredsRead() (NAPI)        读 secure/s3_creds.blob
        ├── HUKS.unwrapDeviceBound()    device-bound 解密
        ├── setS3CredsRuntime(ak, sk)   写 AppState.s3_creds RwLock
        └── s3CredsClearLegacy()        一次性删 SQLite 残留(idempotent)

首次配对:
  Onboard → ShibeiService.setS3Config(cfg)
    ├── napiSetS3Config(json)           写 SQLite(endpoint/region/bucket)
    │                                     + 写 AppState.s3_creds RwLock(ak/sk)
    └── HUKS.wrapDeviceBound + s3CredsWrite  持久化给下次冷启动

同步运行:
  sync_metadata → build_raw_backend()
    ├── sync_state::get(endpoint/region/bucket)  SQLite
    └── app.s3_creds.read()                      RwLock;空则 error.credentialsNotPrimed
```

### 3.2 关键决策

- **只写内存,不写 SQLite**:`build_raw_backend` 从 `app.s3_creds.read()` 读;如果 RwLock 为 None,返回 `error.credentialsNotPrimed`(不应该在正常流程出现——正常 Onboard 或冷启动都会填充)
- **`set_s3_config` NAPI 同时填 RwLock**:避免配对完后要等重启才能 sync 的尴尬
- **`reset_device` 同步清 RwLock**:之前只 `lock_vault()` 清 MK,现在也清 S3 creds,防止重置后旧 AK 还能联网
- **兼容性**:老装机(Phase 4)SQLite 里有 AK/SK 行——`primeS3Creds` 每次启动跑一次 `s3CredsClearLegacy`(SQL DELETE,幂等),一次就清完了

### 3.3 改动点

- `src-harmony-napi/src/state.rs`:`AppState` 加 `s3_creds: Arc<RwLock<Option<(String, String)>>>`;新增 `clear_s3_creds()` 助手
- `src-harmony-napi/src/commands.rs`:
  - `build_raw_backend` 改读 RwLock
  - `set_s3_config` 移除 `store_credentials` 调用;改写入 RwLock
  - `set_s3_creds_only` → `set_s3_creds_runtime`(仅写 RwLock)
  - `reset_device_inner` 追加 `state::clear_s3_creds()`
- `shibei-harmony/entry/src/main/ets/services/ShibeiService.ets`:
  - `primeS3Creds` 改调 `setS3CredsRuntime` + 追加 `s3CredsClearLegacy`
  - `setS3CredsOnly` → `setS3CredsRuntime`(重命名,行为变更)
- NAPI bindings 重新生成(`cargo run -p shibei-napi-codegen`)

## 4. 桌面端:S3 creds → OS Keychain(A+ 方案)

### 4.1 A+ 策略

| 路径 | 行为 |
|---|---|
| **写入(`store_credentials`)** | **严格**——只写 keychain,失败返错 `DbError::InvalidOperation("keystore save failed: ...")`。绝不降级写 SQLite |
| **读取(`load_credentials`)** | **宽容**——先 keychain,空则 fallback SQLite;keychain 平台错误也 fallback(eprintln warn) |
| **删除(`delete_credentials`)** | 两侧都删,best-effort keychain |
| **迁移(Tauri setup)** | 一次性:SQLite 有 creds && keychain 空 → 拷过去 + 删 SQLite 行。失败留 SQLite 由下次启动重试;fallback 路径保证当前会话 sync 可用 |

### 4.2 entry naming

```rust
const SERVICE: &str = "shibei";
const ACCOUNT: &str = "s3-credentials";  // 与 "encryption-master-key" 并列
```

单 entry 存 JSON blob `{"access_key": "...", "secret_key": "..."}`——保证 AK+SK 原子更新,避免写一半挂掉导致 mismatch。

### 4.3 Fallback 生存场景

- **macOS**:钥匙串在被用户手动锁定 / iCloud keychain 权限异常时可能拒绝 `get_secret`。fallback SQLite 让当前会话跑通,next startup 重试
- **Linux(headless / 无 secret-service)**:save 直接错,用户看到"keystore save failed"——必须在有桌面环境的机器上首次配置
- **Windows**:几乎不会失败

### 4.4 改动点

- `crates/shibei-sync/src/os_keystore.rs`:新增 `save_s3_credentials` / `load_s3_credentials` / `delete_s3_credentials`,共用 `S3CredsBlob` JSON 序列化;追加 2 个 `#[ignore]` 真钥匙串测试
- `crates/shibei-sync/src/credentials.rs`:`store_credentials` 改调 keystore;`load_credentials` 走 keystore-first-fallback-SQLite;`delete_credentials` 双删;新增 `migrate_credentials_to_keystore` 供 setup hook 调用
- `src-tauri/src/lib.rs`:`.setup()` 里起 thread 跑一次迁移,与 FTS 初始化并列
- `crates/shibei-sync/src/pairing.rs`:测试 setup 的 `store_credentials` 改成 `seed_legacy_creds` 直接写 sync_state(避免 `cargo test` 弹 macOS 钥匙串授权框挂住)

## 5. 迁移路径

### 5.1 手机老装机(Phase 4 已升级)

- 装上 Phase 4.1 build → 冷启动
- `primeS3Creds` 解 HUKS blob 填 RwLock → 调 `s3CredsClearLegacy` 清 SQLite
- 下次启动走新路径

### 5.2 桌面老装机(SQLite 里有 AK/SK)

- 装上 Phase 4.1 build → Tauri `setup` 跑迁移 thread
- 正常情况:1 次启动完成迁移,SQLite 行被删,keychain 里多一项 `shibei/s3-credentials`
- 异常情况(keystore 暂时不可用):SQLite 行保留,当前会话走 fallback;下次启动自动重试
- macOS 首次会弹一次"Shibei 想要访问钥匙串"的授权框——用户点允许即可

### 5.3 新装机

- 手机:Onboard 扫 QR → `setS3Config` 同时写 HUKS blob + RwLock,SQLite 从不出现 AK/SK
- 桌面:Settings 填表 → `cmd_save_sync_config` → `store_credentials` 直接写 keychain

### 6.1 单元 / 集成测试

- `cargo test -p shibei-sync --lib` 全绿(54/54)

### 6.2 手机端验证(Mate X5 真机)

1. `hdc shell sqlite3 shibei.db` → `config:s3_%_key` 行**不存在**(只剩 bucket/endpoint/region) ✓
2. 冷启动 → 指纹解锁 → 下拉刷新触发 sync → 成功 ✓
3. `aa force-stop` kill 后冷启 → 自动 re-prime → 下拉刷新同步仍成功 ✓
4. `hilog -T shibei` 无 `s3 creds unwrap failed` 错误 ✓

### 6.3 桌面端验证(macOS)

1. `sqlite3 ~/Library/Application\ Support/shibei/shibei.db "SELECT key FROM sync_state WHERE key LIKE 'config:s3_%_key%';"` 返回空 ✓
2. `security find-generic-password -s "shibei" -a "s3-credentials"` 找到 entry ✓
3. 应用内触发一次 sync → `last_sync_at` 更新 + `remote:*:last_seq` cursor 推进 ✓
4. sync 完成后 SQLite 仍无 AK/SK 行(确认 load 路径不回写) ✓

## 7. 未解决 / 后续

- **进程内存攻击**:RwLock / `Credentials` 结构体里的明文没有 zeroize 语义;要防需要把 AK/SK 也换成 `Zeroizing<String>` 并把 rust-s3 `Credentials` 构造成 `&[u8]` 回 zeroize——后续议题
- **桌面 Linux 全链路**:当前设计降级到 SQLite fallback 对 Linux desktop 用户(需 secret-service daemon)不够严格;后续可以考虑加一个用户可见的"钥匙串不可用,请手动初始化"提示
- **Keychain entry 损坏**:用户若手动删了 `shibei/s3-credentials`,load 会 fallback SQLite;SQLite 也空则 sync 失败,需要用户重新在 Settings 保存一次——已知行为,UI 侧可以加友好提示
