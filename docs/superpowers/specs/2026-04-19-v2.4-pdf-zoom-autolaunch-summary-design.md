## 1. PDF 缩放

### 1.1 需求

- PDF 阅读器当前按 fit-to-width 渲染（`scale = containerWidth / pageWidth`），当窗口较宽或页面较宽时，纵向可视内容过少。
- 用户需要自行缩放 PDF（放大看细节 / 缩小看整版），并且：
  - 缩放不破坏标注定位、选区、滚动位置等现有功能。
  - 关闭应用后下次打开同一份 PDF 应保持缩放值。

### 1.2 方案级选择

| 候选 | 取舍 |
|---|---|
| 逻辑倍率乘法（改变渲染 scale） | **选定**。Canvas / TextLayer / Highlight rect 均由 scale 统一驱动，anchor 结构无需变更。 |
| CSS `transform: scale()` | 否。会破坏 hit-test、`getClientRects()` 相对坐标、选区矩形。 |
| 多分辨率预渲染 | 否。实现复杂度不成比例，PDF.js 已支持动态 scale。 |

### 1.3 数据模型

- 新增 `zoomFactor: number`，范围 `0.5 – 4.0`，步进 `0.05`，默认 `1.0`。
- **100% 定义为 fit-to-width**，即 `zoomFactor = 1.0` 等于当前行为；不锚定 PDF 原生 72dpi。
- 最终渲染 scale：`scale = (containerWidth * zoomFactor) / pageInfo.width`。

### 1.4 布局变更

在 `PDFReader` 的 `containerRef` 内新增一层内容包裹 `.content`：

```
<div ref={containerRef} overflow:auto>                     ← 原滚动容器
  <div className={styles.content}                          ← 新增
       style={{ width: `${zoomFactor * 100}%`,
                margin: zoomFactor < 1 ? '0 auto' : '0' }}>
    {pageInfos.map((info, idx) => (
      <div className={styles.pageContainer}
           style={{ aspectRatio: `${info.width} / ${info.height}` }} />
    ))}
  </div>
</div>
```

- `zoomFactor > 1`：`.content` 宽于容器，`overflow-x: auto` 出现水平滚动条。
- `zoomFactor < 1`：`.content` 窄于容器，`margin: 0 auto` 居中显示，两侧留白。
- `zoomFactor = 1`：行为与当前完全一致。
- `.container` 的 CSS 从 `overflow-y: auto` 改为 `overflow: auto`。

### 1.5 渲染与 Anchor 兼容

- **Canvas**：`page.getViewport({ scale })` + HiDPI 视口 `scale * devicePixelRatio`，逻辑不变，仅 scale 值随 `zoomFactor` 变化。
- **TextLayer**：`new TextLayer({ textContentSource, container, viewport })`，使用新的 `viewport` 即可，PDF.js 自动处理字体缩放。
- **Highlight rect**：`getHighlightRects()` 内部用 `range.getClientRects()` 实时查询浏览器布局，rect 自然跟随 scale 变化。**anchor 持久化数据结构不变**（`PdfAnchor.charIndex / length / textQuote`）。
- **页高 / 偏移**：
  ```typescript
  getPageHeights() = pageInfos.map(info =>
    info.height * (containerWidth * zoomFactor / info.width)
  );
  ```
  `getPageOffsets()` 基于新的 heights 重算。

### 1.6 交互

#### 工具条位置

复用 `ReaderView` 的 meta 栏（已随滚动方向自动隐藏）。PDF 模式下在 meta 栏右端渲染缩放控件，取代 HTML 模式下的 🌓 反色按钮位置：

```
[title] [url] [date] .................... [−] [120%] [+] [↺]
```

- `−`：缩小 5%
- `[120%]`：纯 label（当前倍率），v2.4 不支持点击输入
- `+`：放大 5%
- `↺`：重置为 100%
- 边界：`zoomFactor <= 0.5` 时 `−` 置灰；`>= 4.0` 时 `+` 置灰

#### 快捷键

全局监听 `keydown`（跨平台使用 `event.ctrlKey || event.metaKey`）：

| 按键 | 动作 |
|---|---|
| `Ctrl/Cmd +` 或 `Ctrl/Cmd =` | 放大 5% |
| `Ctrl/Cmd -` | 缩小 5% |
| `Ctrl/Cmd 0` | 重置 100% |

过滤条件：
- `e.target` 是 `INPUT / TEXTAREA / [contenteditable]` 时忽略。
- 当前活跃 Tab 必须是 PDF 资料时才响应（通过 active tab + resource_type 判断）。

### 1.7 持久化

扩展 `src/lib/sessionState.ts` 的 `ReaderTab` 类型：

```typescript
type ReaderTab = {
  id: string;
  scrollY?: number;       // HTML 专用
  pdfPage?: number;       // 已有
  pdfFraction?: number;   // 已有
  pdfZoom?: number;       // ★ 新增
};
```

- 缩放变更时立即调用 `updateReaderTab(id, { pdfZoom })` 写入（不 debounce；缩放频率远低于滚动）。
- 打开 PDF 时从 session 读 `pdfZoom`，缺失或越界回落到 `1.0`。
- `sessionState.ts` 的 `version` 字段不升级（新增可选字段向后兼容）。

### 1.8 缩放时的滚动位置保持

缩放会改变 `.content` 总高度，需保持用户阅读焦点：

1. **纵向**：缩放前记录 `scrollFractionRef = scrollTop / scrollHeight`；新布局完成后（`ResizeObserver` 已有 200ms debounce 机制可复用），按 fraction 恢复 `scrollTop`。
2. **横向**：缩放后若 `scrollWidth > clientWidth`，水平居中（`scrollLeft = (scrollWidth - clientWidth) / 2`）；缩回 100% 或更低时 `scrollLeft = 0`。

### 1.9 边界场景

| 场景 | 行为 |
|---|---|
| 窗口 resize 时处于高 zoom | `.content` 宽度按百分比相对容器宽度，自动跟随；`ResizeObserver` 触发 scale 重算，页面重渲染。 |
| 从 > 100% 缩回 ≤ 100% | `scrollLeft` 归零，水平滚动条消失。 |
| Tab 切换（切到另一份 PDF） | 每份资料独立 `pdfZoom`；session 中按 tab id 存储。 |
| 首次打开无 session 记录 | 默认 1.0。 |
| `pdfZoom` 超出合法范围（旧数据 / 手动篡改） | 载入时 clamp 到 `[0.5, 4.0]`，否则回落 1.0。 |
| 深链高亮跳转优先于滚动恢复 | 已有 `initialHighlightId` guard 机制保留；zoom 应用后触发高亮 scrollIntoView。 |

### 1.10 i18n

`reader` namespace 新增：
- `zoomIn` / `zoomOut` / `zoomReset`（按钮 title）
- `zoomPercentage`（格式化 label，无插值则直接用 `{value}%`）

## 2. 开机自启

### 2.1 需求

- 用户希望系统登录后 Shibei 自动启动，但不抢占焦点。
- 双平台（macOS + Windows）都支持。

### 2.2 方案级选择

| 候选 | 取舍 |
|---|---|
| `tauri-plugin-autostart` + 启动时 `window.minimize()` | **选定**。官方插件屏蔽双平台差异，无需引入 tray。 |
| 自写 LaunchAgent / 注册表操作 | 否。重造轮子，难维护。 |
| 引入 System Tray + 后台常驻 | 否。改动量大，超出本版本范围。 |

### 2.3 启动流程

**Rust 侧**（`src-tauri/src/lib.rs` 或 `main.rs`）：

```rust
use tauri_plugin_autostart::MacosLauncher;

tauri::Builder::default()
    .plugin(tauri_plugin_autostart::init(
        MacosLauncher::LaunchAgent,
        Some(vec!["--autostart"]),
    ))
    .setup(|app| {
        let is_autostart = std::env::args().any(|a| a == "--autostart");
        if is_autostart {
            let window = app.get_webview_window("main")
                .ok_or("main window not found")?;
            window.minimize()?;
        }
        Ok(())
    })
    ...
```

**行为**：
- **macOS**：通过 LaunchAgent 注册登录启动，`--autostart` 传入后窗口最小化至 Dock；Dock 图标可见，用户点击 Dock 恢复窗口。
- **Windows**：通过注册表 `HKCU\Software\Microsoft\Windows\CurrentVersion\Run` 注册；启动后窗口最小化至任务栏，点击任务栏恢复。
- 单实例（`tauri-plugin-single-instance`）已存在，用户手动再次启动时会聚焦已有窗口，不会重复打开。

### 2.4 前端 API

`src/lib/autostart.ts`（新文件）：

```typescript
import { enable, disable, isEnabled } from "@tauri-apps/plugin-autostart";

export const getAutoLaunchEnabled = (): Promise<boolean> => isEnabled();
export const setAutoLaunchEnabled = async (on: boolean): Promise<void> => {
  if (on) await enable();
  else await disable();
};
```

### 2.5 Settings UI

**位置**：`Settings/AppearancePage.tsx` 底部新加 `启动` 分区（沿用现有 `<h3>` + `<div passwordSection>` 的分隔模式）。

**UI**：单一 checkbox + 说明文字
- 标签：「开机时自动启动 Shibei（最小化到 Dock / 任务栏）」
- 说明：「下次登录系统时自动打开，不抢占焦点」

**状态加载**：组件挂载时调 `getAutoLaunchEnabled()` 读系统真实状态；不做 localStorage 镜像，避免与系统状态分歧。

**切换逻辑**：
1. 用户点击 checkbox → 调 `setAutoLaunchEnabled(newValue)`
2. 成功 → toast 确认，本地 state 更新
3. 失败（权限拒绝等）→ catch 后 toast 错误，state 回滚至 `getAutoLaunchEnabled()` 当前值

### 2.6 i18n

`settings` namespace 新增：
- `startup`（"启动" / "Startup"）
- `autoLaunch`（"开机时自动启动 Shibei" / "Launch Shibei at startup"）
- `autoLaunchDescription`（说明文案）
- `autoLaunchEnabled` / `autoLaunchDisabled`（toast 文案）
- `autoLaunchFailed`（错误文案）

### 2.7 边界场景

| 场景 | 行为 |
|---|---|
| macOS 首次启用弹权限确认 | 由系统处理，无需 app 侧逻辑。用户拒绝则 `enable()` 报错，UI 回滚。 |
| App 路径变化（卸载重装 / 迁移） | 系统记录失效，`isEnabled()` 返回 false，UI 自动同步为"关闭"，用户需重新勾选。 |
| `--autostart` 通过其他途径传入（非系统自启） | 仍然最小化；这是可接受的副作用，不做区分。 |
| 开发模式（`cargo tauri dev`） | autostart 记录的是 debug binary 路径，卸载插件后失效；仅打包版本测试此功能。 |

## 3. 阅读器摘要显示

### 3.1 需求

- 在阅读器右栏 `AnnotationPanel` 顶部显示资料摘要。
- 内容来源：`description.trim()` 优先，空则取 `plain_text` 前 200 字。
- 两者都空则整块不渲染。
- 添加摘要不得压缩标注列表的可视空间。

### 3.2 方案级选择

| 候选 | 取舍 |
|---|---|
| 摘要放 `scrollArea` 内顶部，`标注 (N)` header 也迁入 | **选定**。摘要随列表滚动，不永久占用可视空间；header 紧邻 list 语义清晰。 |
| 摘要放 ResourceMeta 下的固定区 | 否。永久压缩 ~150px 标注可视空间（~25%）。 |
| 摘要作为 ResourceMeta 悬停 tooltip | 否。发现性差，不符合"摘要是内容的一部分"语义。 |

### 3.3 结构变更

```
<panel>
  <ResourceMeta />                          ← 固定（不变）
  <stickyAnnotationsHeader />               ★ 新增，位于 scrollArea 之上
  <scrollArea>                              ← 原 scrollArea
    <SummarySection />                      ★ 新增，scrollArea 顶部
    <AnnotationsHeader ref={headerRef}/>    ★ 从固定区迁入
    <list>{highlights}</list>
    <NotesList />                           ← 内含"笔记 (N)" header
  </scrollArea>
  <stickyNotesHeader />                     ← 保留（位于 scrollArea 之下）
  <NoteInput />                             ← 固定（不变）
</panel>
```

- `stickyAnnotationsHeader` 放在 `scrollArea` **之前**（作为顶部 sticky，annotations header 向上滚出视口后显示），与 `stickyNotesHeader` 放在 `scrollArea` 之后（作为底部 sticky）形成对称。

### 3.4 SummarySection 实现

内联在 `AnnotationPanel.tsx` 中（与 `NotesList` / `NoteInput` 同文件）：

```tsx
function SummarySection({ resource }: { resource: Resource }) {
  const { t } = useTranslation("annotation");
  const [summary, setSummary] = useState<string | null>(null);

  useEffect(() => {
    const desc = resource.description?.trim();
    if (desc) {
      setSummary(desc);
      return;
    }
    cmd.getResourceSummary(resource.id).then((s) => {
      setSummary(s?.trim() || null);
    });
  }, [resource.id, resource.description]);

  if (!summary) return null;

  return (
    <section className={styles.summarySection}>
      <div className={styles.sectionHeader}>{t("summary")}</div>
      <div className={styles.summaryText}>{summary}</div>
    </section>
  );
}
```

- 复用后端已有命令 `cmd_get_resource_summary`（供 PreviewPanel 使用），前 200 字截断在后端完成。
- `resource.description` 通过 ResourceEditDialog 编辑后，`data:resource-changed` 事件会让 ReaderView 传入新的 `resource` prop，effect 自然重跑。
- 资料切换（id 变化）时 effect 重跑。
- 两者都空时 `summary = null`，整块不渲染。

### 3.5 AnnotationsHeader 迁入 scrollArea

```tsx
<div ref={annotationsHeaderRef} className={styles.sectionHeader}>
  {t("annotations")} ({highlights.length})
</div>
```

- 文案格式：`标注 (6)` / `Annotations (6)`（与 `NotesList` 的"笔记 (N)"视觉统一，与截图一致）
- 原 i18n key `annotationsCount`（"X 条标注"格式）删除，替换为 `annotations` + 括号数量
- 原 `.header` CSS class（含 `border-bottom`）**废弃**，改用 `.sectionHeader`（无 border，依赖上下 margin 分隔）

### 3.6 Sticky Annotations Header

完全复用 `stickyNotesHeader` 的 `IntersectionObserver` 模式：

```tsx
const [annotationsHeaderHidden, setAnnotationsHeaderHidden] = useState(false);
useEffect(() => {
  const header = annotationsHeaderRef.current;
  const root = scrollAreaRef.current;
  if (!header || !root) return;
  const observer = new IntersectionObserver(
    ([entry]) => setAnnotationsHeaderHidden(!entry.isIntersecting),
    { root, threshold: 0 },
  );
  observer.observe(header);
  return () => observer.disconnect();
}, []);

{annotationsHeaderHidden && !notesHeaderHidden && (
  <div
    className={styles.stickyAnnotationsHeader}
    onClick={() => annotationsHeaderRef.current?.scrollIntoView(
      { behavior: "smooth", block: "start" }
    )}
  >
    {t("annotations")} ({highlights.length})
  </div>
)}
```

**Sticky 触发条件**：`stickyAnnotationsHeader` 仅在 annotations header 被向上滚出视口时显示（而非向下，即初始未进入视口时不显示）。通过 `entry.boundingClientRect.top < rootBounds.top` 判别：

```tsx
const observer = new IntersectionObserver(
  ([entry]) => {
    const scrolledAbove =
      !entry.isIntersecting &&
      entry.boundingClientRect.top < (entry.rootBounds?.top ?? 0);
    setAnnotationsHeaderHidden(scrolledAbove);
  },
  { root: scrollAreaRef.current, threshold: 0 },
);
```

避免摘要很长时 annotations header 初始位于视口下方也触发 sticky。

### 3.8 i18n

`annotation` namespace：
- **新增**：`summary`（"摘要" / "Summary"）、`annotations`（"标注" / "Annotations"）
- **移除**：`annotationsCount`（旧"X 条标注"格式）

### 3.9 边界场景

| 场景 | 行为 |
|---|---|
| description 和 plain_text 都为空 | SummarySection 返回 null 不渲染；ResourceMeta 下直接是 `标注 (0)` header（在 scrollArea 内）。 |
| plain_text 未就绪（PDF 前端兜底回填中） | `getResourceSummary` 返回空 → summary 为 null。下次打开时若已回填则展示；不做 v2.4 范围内的主动轮询。 |
| description 被用户编辑为空 | `data:resource-changed` 触发 effect 重跑，取 plain_text 回退或整块消失。 |
| description 超长（数千字手写） | 不截断，尊重用户输入；溢出让 scrollArea 自然滚动。 |
| 摘要文字中包含换行 | `white-space: pre-wrap` 保留原换行。 |
| Summary + Annotations 双 sticky 同时触发 | 仅显示笔记 sticky；见 3.6 约束。 |
