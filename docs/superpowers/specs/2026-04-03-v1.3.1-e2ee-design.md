# v1.3.1 E2EE 设计文档

## 目标

为 S3 云同步增加端到端加密（E2EE），确保 S3 提供商或任何能访问 bucket 的人无法读取用户数据。本地数据保持明文，不在本版本范围内。

## 设计决策摘要

| 决策项 | 选择 | 理由 |
|--------|------|------|
| 威胁模型 | 仅保护 S3 数据 | 本地靠 OS 保护，云端是主要风险面 |
| 密钥管理 | 随机 MK + 密码保护 | 换密码不用重新加密数据 |
| 加密粒度 | 整文件加密 | snapshot.html 通常几 MB，无需分块 |
| 路径加密 | 不加密 | 元信息泄露风险低，路径可读方便调试 |
| 迁移策略 | 一刀切清空重传 | 个人工具设备少，不维护新旧两套解密路径 |
| 加密层位置 | EncryptedBackend 装饰器 | SyncEngine 零改动，加密逻辑集中可测 |
| 关闭加密 | v1.3.1 不支持 | 需全量明文重传+多设备协调，复杂度高需求弱 |

## 加密方案

### 算法

**XChaCha20-Poly1305**（AEAD 认证加密）

- Nonce: 24 bytes，可安全使用随机 nonce（无需计数器管理）
- Poly1305 tag: 16 bytes，提供完整性校验
- Rust crate: `chacha20poly1305`

### 密钥层级

```
用户密码
    ↓ Argon2id (salt 存于 S3 keyring.json)
密码派生密钥 (PDK, 256-bit)
    ↓ 解密 wrapped_master_key
主密钥 (MK, 256-bit 随机生成 via OsRng)
    ↓ XChaCha20-Poly1305 加密所有上传数据
密文
```

### Argon2id 参数

初始参数（OWASP 最低推荐线，个人工具可接受）：

- `m_cost`: 65536 (64 MB)
- `t_cost`: 3
- `p_cost`: 1

参数存储在 `keyring.json` 中，后续版本可通过 keyring version 字段升级参数而无需重新加密数据。

### 加密文件格式

每个上传到 S3 的文件（keyring.json 除外）：

```
[1 byte: version = 0x01]
[24 bytes: random nonce]
[remaining: ciphertext + 16-byte Poly1305 tag]
```

**AAD（Associated Data）**：S3 object key（文件路径）作为 AAD 传入加密，防止文件互换攻击。实现成本几乎为零。

### Verification Hash

MK 生成时同时派生 verification hash：

```
verification_hash = HKDF-SHA256(ikm=MK, info="shibei-verify", length=16)
```

存入 keyring.json。解密 MK 后先验证此 hash：
- 匹配 → 密码正确，MK 有效
- 不匹配 → 明确提示"密码错误或密钥文件被篡改"

### keyring.json

明文存储在 S3 `meta/keyring.json`，由专门函数通过裸 S3Backend 读写（不经过 EncryptedBackend）：

```json
{
  "version": 1,
  "kdf": "argon2id",
  "argon2_params": { "m_cost": 65536, "t_cost": 3, "p_cost": 1 },
  "salt": "<base64, 32 bytes>",
  "wrapped_master_key": "<base64, encrypted MK>",
  "wrapped_master_key_nonce": "<base64, 24 bytes>",
  "verification_hash": "<base64, 16 bytes>"
}
```

## 架构

### EncryptedBackend 装饰器

```rust
pub struct EncryptedBackend<B: SyncBackend> {
    inner: B,
    master_key: Zeroizing<[u8; 32]>,
}

impl<B: SyncBackend> SyncBackend for EncryptedBackend<B> {
    async fn upload(&self, key: &str, data: &[u8]) -> Result<String, BackendError> {
        let encrypted = encrypt(data, &self.master_key, key.as_bytes())?;
        self.inner.upload(key, &encrypted).await
    }

    async fn download(&self, key: &str) -> Result<Vec<u8>, BackendError> {
        let encrypted = self.inner.download(key).await?;
        decrypt(&encrypted, &self.master_key, key.as_bytes())
    }

    async fn list(&self, prefix: &str) -> Result<Vec<ObjectInfo>, BackendError> {
        self.inner.list(prefix).await // 透传
    }

    async fn delete(&self, key: &str) -> Result<(), BackendError> {
        self.inner.delete(key).await // 透传
    }

    async fn head(&self, key: &str) -> Result<Option<ObjectMeta>, BackendError> {
        self.inner.head(key).await // 透传
    }
}
```

SyncEngine 代码零改动，加密逻辑完全封装在装饰器内。

### SyncEngine 构建逻辑

```
AppState 中有 MK?
  → Yes: 构建 S3Backend → 包 EncryptedBackend → 传入 SyncEngine
  → No + head("meta/keyring.json") 存在: 返回 EncryptionRequired 错误
  → No + keyring.json 不存在 + config:encryption_enabled != "true": 明文同步
```

### MK 在 AppState 的存储

```rust
// Tauri managed state
pub struct EncryptionState {
    master_key: Mutex<Option<Zeroizing<[u8; 32]>>>,
}
```

- `None`: 未配置加密 / 未输入密码
- `Some(...)`: 已解锁，可正常加密同步
- `Zeroizing` 包装确保 Drop 时（包括 panic 路径）清零内存
- 随机数生成统一使用 `OsRng`（CSPRNG）

### 加密范围

所有上传到 S3 的文件都加密：

- JSONL 同步日志（含完整业务数据 payload）
- 全量快照 JSON
- snapshot.html 资料快照

唯一例外：`meta/keyring.json`（明文，密钥元信息）。

### S3 文件结构（不变）

```
s3://bucket/
├── meta/
│   └── keyring.json          ← 明文（密钥元信息）
├── sync/
│   └── {device_id}/
│       └── {timestamp}.jsonl ← 加密
├── state/
│   └── snapshot-{ts}.json    ← 加密
└── snapshots/
    └── {resource_id}/
        └── snapshot.html     ← 加密
```

路径保持明文可读，仅文件内容加密。

## 用户流程

### 首次启用加密

```
同步设置页 → 开启"端到端加密" → 弹出密码设置对话框
→ 输入密码 + 确认密码 → invoke cmd_setup_encryption
→ Rust:
    1. 生成 salt(32B) + MK(32B) via OsRng
    2. Argon2id 派生 PDK
    3. XChaCha20 wrap MK，生成 verification_hash
    4. 上传 keyring.json（裸 S3Backend）
    5. 清空 S3 上 sync/ + state/ + snapshots/ 全部旧文件
    6. 本地 sync_log 标记全部 uploaded = 0
    7. 清空 sync_state 中所有 remote:*:last_seq 记录
    8. MK 存入 AppState
    9. 设置 config:encryption_enabled = "true"
    10. 触发完整 sync（全量快照 + snapshot.html，走 EncryptedBackend）
→ 前端收到成功 → 显示已加密状态
```

失败回滚：
- 步骤 4 之前失败：无副作用，直接报错
- 步骤 5-7 中间失败：S3 可能部分清空，下次重试继续清理（幂等）
- 步骤 10 失败：等同正常 sync 失败，下次 sync 补上

### 应用重启解锁

```
应用启动 → 检查 config:encryption_enabled
→ "true": 同步设置页显示"输入加密密码"按钮
→ 用户输入密码 → invoke cmd_unlock_encryption
→ Rust:
    1. 从 S3 下载 keyring.json（裸 S3Backend）
    2. Argon2id 派生 PDK
    3. 解密 wrapped_master_key
    4. 验证 verification_hash
    5. 成功: MK 存入 AppState，sync 恢复正常
    6. 失败: 返回 WrongPassword / KeyringTampered 错误
```

密码输入入口在同步设置页，不等到 sync 失败才弹窗。

### 多设备场景

设备 A 启用加密后，设备 B 下次 sync：

```
SyncEngine 构建 → head("meta/keyring.json") → 发现远端已加密
→ 本地 MK 未就绪 → 返回 EncryptionRequired 错误
→ 前端提示用户输入密码 → cmd_unlock_encryption → MK 就绪
→ 本地清空 sync 进度（步骤 6-7 同首次启用）
→ 重新 sync（从远端拉取加密数据，EncryptedBackend 自动解密）
```

设备 B 不需要重新上传，清空本地同步进度后视为新设备——走 Phase 1.5（导入最新全量快照）+ 增量回放，与正常的新设备加入流程一致。

### 修改密码

```
同步设置页 → "修改加密密码" → 输入旧密码 + 新密码
→ invoke cmd_change_encryption_password
→ Rust:
    1. 下载 keyring.json → 旧密码派生 PDK → 解密 MK → 验证 verification_hash
    2. 新密码派生新 PDK → 重新 wrap 同一 MK
    3. 上传新 keyring.json
    4. 不需要重新加密任何数据（两层密钥的价值）
```

## 错误处理

新增错误类型，作为 `SyncError` 变体：

| 错误 | 触发场景 | 用户提示 |
|------|---------|---------|
| `EncryptionRequired` | 远端已加密但本地 MK 未就绪 | "远端已启用加密，请输入加密密码" |
| `WrongPassword` | verification_hash 不匹配 | "密码错误" |
| `KeyringTampered` | keyring.json 格式异常或解密+验证均失败 | "密钥文件可能被篡改，请检查" |
| `KeyringNotFound` | 本地标记已加密但 S3 上 keyring.json 丢失 | "密钥文件丢失，无法同步" |
| `DecryptionFailed` | 数据文件解密失败（格式错/tag 校验失败） | "数据解密失败，文件可能损坏" |

## 前端变化

### 同步设置页

- **加密开关**：未启用时显示"启用端到端加密"开关 + 明文警告
- **密码设置对话框**：密码 + 确认密码，最少 8 字符
- **密码输入入口**：已配置加密但 MK 未就绪时，显示"输入加密密码"按钮
- **修改密码**：已启用加密时，显示"修改加密密码"入口

### Sidebar 同步状态

扩展现有同步状态指示：

- 未配置加密：保持现状
- 已加密 + MK 就绪：正常同步状态 + 锁图标
- 已加密 + MK 未就绪：显示需输入密码提示

## 新增 Rust 依赖

| Crate | 用途 | 理由 |
|-------|------|------|
| `chacha20poly1305` | AEAD 加密 | XChaCha20-Poly1305，纯 Rust，轻量 |
| `argon2` | 密码派生 | Argon2id KDF |
| `hkdf` + `sha2` | verification_hash 派生 | HKDF-SHA256 |
| `rand` | 安全随机数 | OsRng，显式声明 |
| `zeroize` | 内存清零 | Zeroizing wrapper for MK |

## 测试策略

### 单元测试（Rust）

- **crypto 模块**：encrypt → decrypt 往返；错误 nonce/AAD/密钥 → 解密失败
- **keyring 模块**：生成 → 序列化 → 反序列化往返；密码正确解锁；密码错误拒绝；verification_hash 验证
- **EncryptedBackend**：配合 MockBackend，上传数据是密文、下载正确解密、list/delete/head 透传

### 集成测试（Rust）

- 完整迁移流程：明文 sync → 启用加密 → 验证 S3 上全为密文 → 新设备用密码解锁 → 拉取并解密成功
- 改密码：旧密码失败、新密码成功

### 前端测试

- 同步设置页：加密开关渲染、密码对话框、密码不匹配提示
- 解锁流程：MK 未就绪时的 UI 状态

## 不在 v1.3.1 范围

- 关闭加密（需全量明文重传 + 多设备协调）
- 密钥轮换（更换 MK，需重新加密所有数据）
- 本地数据加密
- 生物识别解锁
- 恢复码 / 密钥托管
