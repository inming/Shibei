# v1.4 — 搜索设计文档

## 目标

让资料库中的资料可被高效检索。分两期实现：第一期覆盖元数据搜索，第二期扩展到快照全文搜索。

## 范围

### 第一期（本次实现）

元数据搜索：标题、URL、描述、高亮文本（`highlights.text_content`）、评论内容（`comments.content`）。

### 第二期（未来）

快照全文搜索：保存快照时提取 HTML 纯文本，写入同一套 FTS5 索引。复用第一期的全部基础设施。

### 不在范围

- 搜索建议 / 自动补全
- 语义搜索 / 向量搜索
- 搜索历史记录
- 按标注内容搜索（高亮文本已覆盖，指的是按标注颜色等属性搜索）

## 后端设计

### FTS5 虚拟表

使用 SQLite FTS5 + trigram tokenizer。trigram 按 3 字符滑窗索引，天然支持中文子串匹配，无需外部分词器。

**前置验证**：rusqlite 0.34 bundled 编译的是 SQLite 3.46.0，默认启用 `SQLITE_ENABLE_FTS5`，支持 trigram tokenizer。实现第一步需写最小测试验证 FTS5 trigram 在当前环境下可用。

**trigram 固有限制**：trigram 按 3 字符滑窗索引，查询词必须 >= 3 字符才能匹配。这意味着中文 1-2 字查询（如"AI"、"算法"）无法通过 FTS5 匹配。前端最小输入限制（3 字符）是由 tokenizer 特性决定的，而非 UX 选择。这是当前方案的已知 trade-off，未来可通过切换到支持更短 token 的 tokenizer（如 unicode61 + 自定义中文分词）来改善。

**Migration `005_search_index.sql`**：

```sql
CREATE VIRTUAL TABLE IF NOT EXISTS search_index USING fts5(
    resource_id UNINDEXED,
    title,
    url,
    description,
    highlights_text,
    comments_text,
    tokenize='trigram'
);
```

- `resource_id`：关联字段，标记 `UNINDEXED` 不参与搜索
- `highlights_text`：该资料所有高亮的 `text_content` 拼接（换行分隔）
- `comments_text`：该资料所有评论的 `content` 拼接（换行分隔）

### 索引维护

所有 FTS 索引更新封装在 `db/search.rs` 模块中。

#### 核心函数

```rust
/// 重建单个资料的 FTS 索引行
pub fn rebuild_search_index(conn: &Connection, resource_id: &str) -> Result<(), DbError>

/// 删除单个资料的 FTS 索引行
pub fn delete_search_index(conn: &Connection, resource_id: &str) -> Result<(), DbError>

/// 重建所有资料的 FTS 索引（首次 migration 或同步后批量使用）
pub fn rebuild_all_search_index(conn: &Connection) -> Result<(), DbError>
```

`rebuild_search_index` 的逻辑：
1. 读取 `resources` 表中该 resource 的 title、url、description
2. 查询该 resource 所有未删除的 highlights，拼接 `text_content`
3. 查询该 resource 所有未删除的 comments，拼接 `content`
4. DELETE + INSERT 到 `search_index`（FTS5 不支持 UPDATE 部分列）

#### 触发时机

| 操作 | FTS 动作 |
|------|---------|
| `create_resource` | `rebuild_search_index` |
| `update_resource`（标题/描述） | `rebuild_search_index` |
| `delete_resource`（软删除） | `delete_search_index` |
| `restore_resource` | `rebuild_search_index` |
| `purge_resource` | `delete_search_index` |
| `create_highlight` / `delete_highlight` | `rebuild_search_index`（对应 resource） |
| `create_comment` / `update_comment` / `delete_comment` | `rebuild_search_index`（对应 resource） |
| 同步 apply 完成 | 对受影响的 resource_id 逐个调用 `rebuild_search_index`（同步引擎 apply 时已知变更列表） |

#### 首次 migration

`005_search_index.sql` 只创建虚拟表。应用启动时通过 `sync_state` 表检查标志位 `config:fts_initialized`，若未初始化则调用 `rebuild_all_search_index` 填充已有数据并写入标志位。这避免了"FTS 表为空"作为判定条件的问题——空资料库本身 FTS 表就是空的，不应每次启动都重复检测。

### 搜索命令

**Tauri Command**：`cmd_search_resources`

```rust
#[tauri::command]
pub fn cmd_search_resources(
    pool: State<DbPool>,
    query: String,
    folder_id: Option<String>,     // None = 全部资料
    tag_ids: Vec<String>,          // 空 = 不过滤标签
    sort_by: String,               // "created_at" | "annotated_at"
    sort_order: String,            // "asc" | "desc"
) -> Result<Vec<Resource>, String>
```

返回标准 `Vec<Resource>`，与现有列表接口一致。前端自行根据搜索词判断高亮位置。

**SQL 查询逻辑**：

```sql
SELECT r.*
FROM search_index si
JOIN resources r ON r.id = si.resource_id
WHERE search_index MATCH ?1
  AND r.deleted_at IS NULL
  AND (r.folder_id = ?2 OR ?2 IS NULL)
  -- tag 过滤通过 EXISTS 子查询
ORDER BY ...
```

其中 `?1` 为转义后的查询词（双引号包裹）。

**不设 `matched_fields`**：后端只返回 `Vec<Resource>`（与现有列表接口一致）。前端拿到搜索词后，直接用 `includes` 检查标题/评论/高亮文本是否包含查询词来决定高亮位置。这避免了 FTS5 匹配逻辑与简单 `contains` 检查在边界情况（大小写、Unicode 标准化）下的不一致。

**输入转义**：FTS5 MATCH 语法包含特殊字符（`"`, `*`, `AND`, `OR`, `NOT`, `NEAR` 等）。搜索函数必须对用户输入做转义处理——将整个查询词用双引号包裹（`"用户输入"`），内部的双引号转义为 `""`。这确保用户输入始终被视为字面量短语搜索，不触发 FTS5 操作符语法。

## 前端设计

### 搜索框

位于 ResourceList 顶部，排序控件上方。

```
┌─────────────────────────────┐
│  🔍 搜索...            ✕    │  ← 搜索输入框
├─────────────────────────────┤
│  排序: 创建时间 ↓           │  ← 现有排序控件
├─────────────────────────────┤
│  资料项 1                   │
│  资料项 2                   │
│  ...                        │
└─────────────────────────────┘
```

- 输入少于 3 字符：不触发搜索，显示原列表
- 输入 >= 3 字符：300ms debounce 后调用 `cmd_search_resources`
- 清空输入框（点击 ✕ 或删除所有文字）：恢复原列表
- 搜索在当前文件夹 + 标签筛选范围内进行

### 搜索触发逻辑

`useResources` hook 增加 `searchQuery` 参数：

```typescript
function useResources(
    folderId: string | null,
    sortBy: SortBy,
    sortOrder: SortOrder,
    searchQuery: string,        // 新增
    selectedTagIds: string[]    // 从 ResourceList 移到 hook 参数
)
```

- `searchQuery` 为空或 < 3 字符：走现有 `listResources` / `listAllResources` 逻辑
- `searchQuery` >= 3 字符：调用 `cmd_search_resources(query, folderId, tagIds, sortBy, sortOrder)`

### 匹配高亮

#### ResourceList 中

- 标题命中时，用 `<mark>` 标签高亮匹配的文字片段
- 其他字段（URL、描述等）命中时列表项外观不变，仅在 PreviewPanel 中体现

#### PreviewPanel 中

- 选中某个搜索结果资料后，PreviewPanel 正常显示资料信息
- 高亮文本和评论内容中匹配的文字用 `<mark>` 标签高亮
- 无搜索词时不做高亮处理

### 状态管理

搜索状态由 `Layout.tsx`（LibraryView）管理：

```typescript
const [searchQuery, setSearchQuery] = useState("");
```

传递给 ResourceList 作为 prop。ResourceList 内部维护输入框的即时值和 debounce 后的搜索词。

### 事件订阅

搜索结果需要响应数据变更事件刷新：
- `data:resource-changed` — 资料增删改
- `data:annotation-changed` — 高亮/评论增删改（影响搜索索引）
- `data:tag-changed` — 标签变更（影响标签过滤）
- `data:sync-completed` — 同步完成后重新搜索

这些事件 `useResources` 已经订阅，搜索模式下同样触发重新查询即可。

## 数据流

```
用户输入搜索词
    ↓
ResourceList（debounce 300ms，>= 3 字符）
    ↓
useResources hook → cmd_search_resources(query, folderId, tagIds, sortBy, sortOrder)
    ↓
Rust: FTS5 MATCH query → JOIN resources → 过滤 folder/tags → 排序
    ↓
返回 Vec<Resource>
    ↓
ResourceList 渲染（前端用搜索词 includes 检查标题，命中则高亮）
    ↓
用户选中资料 → PreviewPanel 渲染（前端用搜索词 includes 检查评论/高亮文本，命中则高亮）
```

## 软删除与 FTS 索引的生命周期

```
create_resource → rebuild_search_index（建索引）
    ↓
delete_resource（软删除） → delete_search_index（删索引，快照保留）
    ↓
restore_resource → rebuild_search_index（重建索引）
    ↓
purge_resource（90天后物理清理） → 此时 FTS 行已在软删除时删除，无需额外操作
```

软删除时即删除 FTS 行，确保已删除资料不出现在搜索结果中。`purge_resource` 正常情况下 FTS 行已在软删除时删除，无需额外操作。但 `delete_search_index` 本身是幂等的（DELETE 不存在的行不报错），因此 purge 流程中可安全调用作为防御措施。

---

## useResources hook 接口变更影响

`useResources` 新增 `searchQuery` 参数，`selectedTagIds` 从 ResourceList 客户端过滤移到 hook 参数（搜索模式下需传给后端）。

当前调用方仅 `Layout.tsx`（LibraryView），影响范围可控。变更后标签过滤统一由后端处理：给 `listResources` / `listAllResources` 也加 `tag_ids` 参数，搜索与非搜索模式走同一过滤逻辑（OR：资料包含任一选中标签即匹配），消除双路径不一致风险。
