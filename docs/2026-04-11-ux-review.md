# 拾贝 (Shibei) UX 体验分析报告

## 一、产品定位与核心体验评估

拾贝是一款「个人只读资料库」桌面应用，核心用户场景是：**网页快照收集 → 阅读 → 标注 → 回顾检索**。这是一个典型的「知识采集-加工-回溯」闭环。

**整体评分：7.5/10** — MVP 功能完整，交互细节扎实（键盘导航、拖拽、深色模式、i18n），但在信息密度、操作流畅度、认知负荷方面有提升空间。

---

## 二、优势分析（做得好的地方）

### 1. 布局架构合理
三栏布局（Sidebar | ResourceList | PreviewPanel）符合「浏览型」应用的主流范式（Finder、Zotero、Bear），用户零学习成本。Tab 式阅读器允许同时打开多篇资料对比阅读，这是正确的架构决策。

### 2. 标注系统技术扎实
- 双锚点系统（textPosition + textQuote + Bitap 模糊匹配）解决了 DOM 变化后高亮丢失的核心痛点
- 5 色高亮 + 评论 + 笔记的三层标注体系覆盖了从「标记」到「思考」的完整需求
- Markdown 评论支持让深度标注成为可能

### 3. 交互细节完善
- 拖拽约束（最小宽度、窗口缩小自适应、持久化）
- 键盘导航覆盖全面（方向键、Enter、Delete、Escape）
- 多选操作（Cmd+Click 切换、Shift+Click 范围选择）
- 搜索支持 IME 组合输入（中文输入法友好）
- 右键菜单上下文感知

### 4. 数据安全意识强
- 软删除 + 回收站 + 确认对话框的三层保护
- 快照文件不修改原文件
- E2EE 加密同步

## 三、体验问题分析与改进建议

### 问题 1：收集到阅读的路径断裂

**现状**：用户通过浏览器插件保存网页后，需要切换到桌面应用 → 找到文件夹 → 在列表中找到刚保存的资料 → 双击打开。这个路径有 3-4 步跳转。

**痛点**：保存后没有即时反馈引导用户去阅读，「收集」和「消化」是割裂的两个场景。

**建议**：
- 插件保存成功后，提供「在拾贝中打开」的可点击链接（通过 deep link `shibei://open/resource/{id}`）

---

### 问题 2：资料列表信息密度不足

**现状**：每个资料项只显示标题 + 域名 + 日期，高度约 60-70px。对于拥有数百条资料的用户，滚动效率较低。

**痛点**：无法快速扫描辨识资料，特别是同一网站的多篇文章标题相似时。

**建议**：
- 提供「紧凑模式」和「详细模式」切换（紧凑模式只显示标题+域名，行高降至 36-40px）
- 在列表项中显示标签色点（小圆点，不占文字空间），提供视觉分类线索
- 考虑增加网站 favicon 显示（SingleFile HTML 中通常包含 favicon 数据），提高视觉辨识度
- 显示标注数量（如 "3 条高亮"），帮助用户识别已深度阅读的资料

---

### 问题 3：预览面板与阅读器职责重叠

**现状**：PreviewPanel 显示标题/URL/标注列表，AnnotationPanel 也显示类似信息。用户在预览面板看到标注后点击进入阅读器，会看到几乎相同的信息。

**痛点**：认知冗余 — 两个面板在呈现相同数据时缺乏差异化的价值主张。

**建议**：
- 预览面板聚焦「概览」：显示资料摘要（description 或正文前 200 字）、标签、文件夹路径、标注统计（N 条高亮、M 条评论），而不是完整标注列表
- 阅读器标注面板聚焦「操作」：保持完整的标注列表 + 编辑能力

---

### 问题 4：标注操作的发现性不足

**现状**：高亮操作需要选中文本 → 右键 → 选颜色。这是一个三步操作，且右键菜单是隐藏交互。

**痛点**：新用户不知道如何高亮，也不知道可以给高亮添加评论。整个标注能力的可发现性依赖用户主动探索。

先维持现状，自动出菜单可能打扰阅读，有的人喜欢勾选阅读

---

### 问题 5：搜索体验可以更强

**现状**：搜索框在 ResourceList 顶部，只在资料列表范围内生效。搜索结果显示高亮标题和「正文匹配」标签。

**痛点**：
- 无法搜索标注内容（评论/笔记），而这往往是用户自己写的、最有回忆线索的内容
- 搜索结果没有正文片段预览，用户不知道为什么某条资料被匹配
- 无法全局搜索（需要先选中正确的文件夹）

**建议**：
- 搜索结果增加匹配上下文片段（snippet），显示匹配关键词前后各 30-50 字
- 全局搜索模式：当在「全部资料」视图时搜索应覆盖所有文件夹（根据 CLAUDE.md 描述目前应该已支持，但需确认 UX 是否清晰传达了这一点）
- 将标注文本（highlights_text/comments_text）的匹配也在搜索结果中标识（如 "标注匹配" 标签），帮助用户理解为什么被命中
- 考虑增加全局快捷键（Cmd+K / Cmd+F）聚焦搜索框

---

### 问题 6：文件夹与标签的使用场景区分不清晰

**现状**：资料可以放入文件夹（单一归属），也可以打标签（多对多）。但 UI 上标签过滤和文件夹导航是并列的，用户不清楚何时用哪个。

**痛点**：分类体系的认知负荷 — 两种组织方式的差异和协作关系不直观。

**建议**：
- 在 Sidebar 中用视觉层次区分：文件夹是「位置」（在哪里），标签是「属性」（是什么）。可以用不同的图标风格强化区分
- 标签过滤目前是 OR 逻辑（包含任一即匹配），考虑是否需要支持 AND 逻辑（用户可能期望「同时包含 A 和 B 标签」的资料）
- 在资料列表中直接显示标签色点，让用户在浏览时就能看到标签分布

---

### 问题 7：回收站体验偏弱

**现状**：回收站是 Sidebar 底部的一个按钮，点击后替换资料列表显示已删除项。

**痛点**：
- 软删除 90 天自动清理的规则用户不可见，可能导致误以为回收站会永久保留
- 没有按删除时间排序或分组的视觉提示

**建议**：
- 回收站内显示每条资料的「剩余保留天数」（如 "还剩 45 天"）
- 回收站顶部显示提示文案：「删除的资料将在 90 天后永久移除」
- 支持批量恢复（多选 + 恢复按钮）

---

### 问题 8：同步状态的透明度不足

**现状**：SyncStatus 组件在 Sidebar 底部显示同步状态（idle/syncing/success/error）。

**痛点**：用户不了解同步的范围和进度，出现冲突时没有可视化提示。

**建议**：
- 同步进行中显示进度提示（如 "正在同步 3/15 条变更"）
- 上次同步时间显示为相对时间（"5 分钟前"），而非绝对时间
- 同步错误时提供可操作的错误信息（"连接 S3 失败，点击重试" 而非纯文本错误）

---

### 问题 9：阅读器的沉浸感可以加强

**现状**：阅读器 Tab 包含顶部元信息栏 + iframe 内容 + 右侧标注面板。

**痛点**：
- 顶部元信息栏始终占用空间，对于长篇阅读是干扰
- 无法全屏阅读（隐藏标注面板）
- 没有阅读进度指示

**建议**：
- 元信息栏在向下滚动时自动收起（sticky header → auto-hide），向上滚动或移到顶部时重新显示
- 在阅读器顶部或滚动条位置显示阅读进度条（细线即可）
- 标注面板可折叠为窄条（只显示高亮色条，不显示文字），悬浮展开

## 四、视觉设计建议

### 1. 字体层级
当前只有 12/14/16px 三级，建议增加 20-24px 的大标题尺寸用于阅读器元信息中的资料标题，增强视觉焦点。

### 2. 间距节奏
当前间距系统（4/8/12/16/24px）覆盖基本需求，但组件间的留白略显紧凑。建议 Section 之间（如文件夹树与标签过滤器之间）使用 24-32px 间距，增加呼吸感。

### 3. 空状态设计
建议为以下空状态提供引导性内容：
- 全新用户（无资料）：引导安装浏览器插件
- 搜索无结果：提示调整关键词
- 文件夹为空：提示拖拽资料到此
- 标注面板为空：提示选中文字进行高亮

### 4. 颜色可及性
高亮色在深色模式下的对比度需要验证，特别是 Yellow（`#F9A825`）在深色背景上的文字可读性。建议对每种高亮色在两种模式下都做 WCAG AA 对比度检查。

## 五、优先级排序

| 优先级 | 改进项 | 影响面 | 实现复杂度 |
|--------|--------|--------|-----------|
| P0 | 搜索结果上下文片段 + 匹配类型标签 | 搜索可用性 | 中 |
| P0 | 预览面板差异化（概览 vs 操作） | 信息架构清晰度 | 中 |
| P1 | 阅读器沉浸感（meta 栏 auto-hide + 进度条 + 面板折叠） | 阅读体验 | 中 |
| P1 | 空状态引导设计 | 新用户体验 | 低 |
| P1 | 资料列表信息密度（标签色点 + 标注数量） | 浏览效率 | 低 |
| P2 | 收集到阅读的路径优化（插件 deep link） | 使用流畅度 | 中 |
| P2 | 回收站天数提示 + 批量恢复 | 数据安全感知 | 低 |
| P2 | Sidebar 文件夹/标签视觉区分 | 认知负荷 | 低 |
| P3 | 同步进度透明化 | 信任感 | 中 |
| P3 | 知识回顾机制 | 长期留存价值 | 高 |

---

## 六、总结

拾贝作为 MVP 产品，功能完备度和技术实现质量都很高。核心的「收集-阅读-标注」闭环已经打通。主要的体验提升方向集中在两个维度：

1. **增强信息效率** — 搜索片段、预览面板差异化、列表信息密度
2. **提升阅读体验** — 元信息栏 auto-hide、阅读进度、标注面板折叠
